## Step 4 产物：最终调研提示词

```markdown
# 调研任务：A 股自动盯盘 AI 助手 · 产品立项调研

## 项目定位

7×24 帮我盯 A 股的 AI 助手：自选股 Dashboard + 自然语言智能选股
+ 早盘简报 + 异动预警。给散户/兼职炒股的程序员用。

## 📁 调研文件输出路径

`<当前项目目录>/specs/research/`

```
research/
├── 01-产品形态.md
├── 02-数据来源.md
├── 03-开源项目.md
├── 04-实现方案.md
└── 05-决策汇总.md
```

## 调研主题 1：产品形态

围绕 A 股自动盯盘 AI 助手，需要回答这些问题。每个都要有可视证据
（截图描述 / 链接 / 真实产品名）：

1. 同类产品盘点：参考 同花顺问财 / 雪球 / 富途 / TradingView，
   在中国大陆市场至少找 10 个 A 股看盘/盯盘/选股类产品。
2. 信息架构：他们首页长什么样？主导航？自选股、行情、推送各自怎么组织？
3. 交互模式：自选股管理、智能选股、异动推送，业内常见交互套路？
4. AI 化案例：已有的 AI 选股/AI 盯盘产品（同花顺问财、AI 助手等）形态？
   做到什么程度？
5. 推送形态：早盘简报/异动预警类业内怎么呈现？（飞书卡片？公众号？App push？）

产出：`01-产品形态.md` —— 产品全景图 + UI/交互参考库

## 调研主题 2：数据来源

围绕 A 股行情数据，能怎么拿？所有可行渠道列出来。

1. 全市场盘点：开源（AKShare、Tushare）、付费、券商接口、爬虫……所有可行渠道
2. 每个渠道的事实：
   - 覆盖范围（A 股全量 / ETF / 港美股）
   - 频率（日线 / 分钟 / Tick）
   - 实时性（延迟几秒）
   - 限频与配额
   - 价格档位
   - 注册门槛（身份证 / 信用卡 / 营业执照）
   - 商用 / 二次分发的合规边界
3. 个人开发者主流选择：实际跑起来的最常用哪几个？为什么？
4. 推荐方案：最推荐用什么 + 备用切换方案

产出：`02-数据来源.md` —— 数据源对照大表 + 推荐方案 + 风险点

## 调研主题 3：开源项目

GitHub / Gitee 上做同类事情的开源项目至少找 10 个。

1. 可参考项目盘点
   关键词参考：A股监控 / 股票 dashboard / 量化交易 / 自选股 /
   stock alert / stock dashboard / quant trading
2. 每个项目的事实：
   - Star / 最近 commit / License
   - 用什么技术栈
   - 做了什么（核心能力）
   - 没做什么（短板）
   - 我能复用什么模块
   - 集成难度（直接 fork / 拆模块 / 仅参考）
3. 生态拼图：哪个项目擅长哪一块
4. 推荐组合：基于"拆模块用"的意图，应该 fork / 借鉴哪几个？

产出：`03-开源项目.md` —— 项目清单 + 复用矩阵 + 初版代码源建议

## 调研主题 4：实现方案

把系统拆成 6 个模块，每个模块给业内主流方案 + 推荐：

| 模块 | 调研维度 |
|------|---------|
| 数据接入 | 接收 + 清洗 + 存储的主流方案 |
| 存储层 | 历史行情存哪 / 实时数据缓存 / 用户数据 |
| 业务逻辑 | 自选股 / 选股引擎 / 异动检测 / 简报生成 各自怎么实现 |
| 前端 UI | Web 框架 / 组件库 / 实时行情图表库 / 实时通信 |
| 推送通道 | 飞书 / 微信 / 钉钉 / Telegram，接入难度+成本 |
| 部署运维 | 本地 / 云服务器 / Serverless，每月预算估算 |

每个模块产出：
- 业内主流 2-3 个方案 + 各自优劣
- 推荐用什么 + 理由（要写"为什么不用 X"）
- 大致工作量预估

产出：`04-实现方案.md` —— 方案对比 + 推荐技术栈 + 架构图（Mermaid）

## 最终决策汇总：`05-决策汇总.md`

1. 产品形态决策：长什么样（首页结构 + 功能形态）
2. 数据源决策：用什么 + 切换备份
3. 开源复用决策：fork / 借鉴哪几个项目
4. 技术栈决策：6 模块各自选什么
5. 架构总图：一张端到端架构图（Mermaid）
6. 成本估算：每月预算
7. 风险：最该警惕的坑
8. 冲突信息处理：列出前 4 份里所有 [冲突 ⚠️] 的事实点 + 取舍理由

## 🔑 产出质量要求

1. 每个结论必须有事实支撑——链接 / 项目名 / 数据
2. 每个推荐必须给理由——"为什么选 X，为什么不选 Y"
3. 不知道就说不知道，不要瞎编

## 🔀 双路调研规范

每个 sub-agent 必须同时用两路搜索引擎，交叉验证：

- **路 A**：Claude 内置 `WebSearch` / `WebFetch`
  （偏海外信源、GitHub、英文官方文档）
- **路 B**：muyu-search-mcp `web_search` / `web_fetch` / `web_map`
  （偏中文信源、国内产品、知乎、国内监管合规）

**muyu-search-mcp 调用约束**：它有 planning 硬门禁，必须先走完
`plan_intent → plan_complexity → plan_sub_query → plan_search_term →
plan_tool_mapping → plan_execution` 才能调 `web_search`，否则会被拒绝。

**事实标注**：每条事实/数据后标来源：
`[WebSearch]` / `[muyu]` / `[双路 ✅]` / `[单源 ⚠️]` / `[冲突 ⚠️ 见下]`

两路冲突时**不要替我裁决**，原样标出 + 列双方来源。

## 并行执行

用 Claude Code 原生 **Task 工具**，在**同一条消息中发起 4 个 Task 调用**。

### Agent 派发清单

- Agent 1：产品形态调研 → `specs/research/01-产品形态.md`
- Agent 2：数据来源调研 → `specs/research/02-数据来源.md`
- Agent 3：开源项目调研 → `specs/research/03-开源项目.md`
- Agent 4：实现方案调研 → `specs/research/04-实现方案.md`

### 4 个 Agent 全部返回后

主 Claude 串行执行收尾：阅读 4 份产物 → 综合写 `05-决策汇总.md`。
这一步不并行——依赖前 4 份产物，需要全局视角。

## 下一步（三段链路，不要跳步）

调研完成（5 份 md 落地）后，按以下顺序推进：

### ① 用 brainstorming Skill 收敛 MVP 方向
> 请仔细阅读 specs/research/ 下的 5 份调研，然后用 Superpowers brainstorming
> Skill 帮我收敛 MVP 方向。必须帮我确定 11 个问题（这是写 PRD 的输入）：
>
> 【为什么/做什么/不做什么】
> 1. 业务目标 + 产品目标 + Non-Goals
> 2. 1-2 个主画像 + 反画像
> 3. 3-5 条核心用户故事（INVEST 格式）
> 4. MVP 功能列表 + Out-of-Scope 清单
> 5. 每个核心功能的关键边界条件
>
> 【怎么算做好了】
> 6. 非功能需求（必须用数字，不要"快"、"友好"等形容词）
> 7. 每个用户故事的验收标准（Given/When/Then）
> 8. 优先级（MoSCoW）
> 9. 北极星指标 + 关停线
>
> 【还不确定的】
> 10. 至少 3 个 Open Questions
> 11. 关键依赖与约束
>
> ⚠️ 不要聊"用 React 还是 Vue"——技术选型留给后面 spec-kit /plan 环节。

### ② 用 prd-writer Skill 写 PRD（极简版）
> 基于 brainstorming 的收敛结果和 specs/research/ 下的调研文件，
> 用 prd-writer Skill 帮我写 PRD（产出到 specs/prd.md）。
>
> ⚠️ 如果 specs/research/06-架构基线决策.md 存在：
> PRD 需新增"架构基线"小节，搬运 06 的复用矩阵。

### ③ 用 GitHub Spec-Kit 转可执行 spec（每个 Must-have 各跑一次）
> 实现 specs/prd.md 中的 <F2 回测引擎>，跑全套 spec-kit 流程：
> /speckit.specify → /speckit.plan → /speckit.tasks → /speckit.implement

⚠️ **不要跳步 + 不要把 PRD 整篇喂给 /speckit.specify**：
- 跳脑暴 → PRD 必然发散
- 跳 PRD → spec 缺业务上下文
- 整篇喂 specify → context 撑爆，要按功能拆条跑（这是 spec-kit 工具不会自动遵守的工作流约束）
```
