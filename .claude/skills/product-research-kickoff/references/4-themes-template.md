## 使用说明

把 [socratic-questions.md](socratic-questions.md) 抽取的 6 个骨架变量代入下面的占位符：

| 占位符 | 替换为 |
|--------|--------|
| `<项目定义>` | 骨架变量 1（项目一句话定义） |
| `<市场>` | 骨架变量 2（目标市场） |
| `<对标 1>` `<对标 2>` `<对标 3>` | 骨架变量 3（对标竞品） |
| `<关键资源>` | 骨架变量 4（关键资源类型） |
| `<开源意图>` | 骨架变量 5 |
| `<模块 1>` ~ `<模块 6>` | 骨架变量 6（6 模块清单） |

## 主题 1：产品形态

```markdown
## 调研主题 1：产品形态

围绕 <项目定义>，需要回答这些问题。每个都要有可视证据
（截图描述 / 链接 / 真实产品名）。

1. 同类产品盘点：参考 <对标 1> / <对标 2> / <对标 3>，
   在 <市场> 市场至少找 10 个同类产品。
2. 信息架构：他们首页长什么样？主导航有哪些？核心功能各自怎么组织？
3. 交互模式：核心操作（创建/管理/触发）业内常见的交互套路是什么？
4. AI 化案例：已有的 AI 化同类产品形态是什么？做到什么程度？
5. 推送/触达形态：业内怎么把结果递达用户？（卡片？图文？通知？邮件？）

产出：`01-产品形态.md` —— 产品全景图 + UI/交互参考库
```

**填充技巧**：
- 第 1 问"至少 10 个"是硬指标，避免 sub-agent 偷懒只找 3 个
- 第 5 问要根据项目形态调整（B 端工具问"嵌入哪个工作流"、C 端问"App push 还是公众号"）

## 主题 2：关键资源

根据骨架变量 4（关键资源类型）选择对应模板：

### 4A：数据型（行情/公开数据集/UGC 数据）

```markdown
## 调研主题 2：数据来源

围绕 <项目定义>，<关键资源> 怎么拿？所有可行渠道列出来。

1. 全市场盘点：开源、付费、官方接口、爬虫……所有可行渠道
2. 每个渠道的事实：
   - 覆盖范围（全量 / 部分 / 区域）
   - 频率（实时 / 日级 / 历史回填）
   - 实时性（延迟）
   - 限频与配额
   - 价格档位
   - 注册门槛（身份证 / 信用卡 / 营业执照）
   - 商用 / 二次分发的合规边界
3. 个人开发者主流选择：实际跑起来的最常用哪几个？为什么？
4. 推荐方案：最推荐用什么 + 备用切换方案

产出：`02-数据来源.md` —— 数据源对照大表 + 推荐方案 + 风险点
```

### 4B：API 型（第三方服务/模型 API/SaaS）

```markdown
## 调研主题 2：API/服务来源

围绕 <项目定义>，<关键资源> 有哪些供应商可选？

1. 供应商盘点：至少 5 家可对比
2. 每家的事实：
   - 价格档位（含免费额度）
   - SLA / 可用性
   - 限额（QPS / 月度配额）
   - 区域可用性（含国内访问情况）
   - 合规（数据驻留 / 隐私 / 行业资质）
   - 切换成本（API 兼容性）
3. 个人开发者主流选择：常用哪几家？为什么？
4. 推荐方案：主选 + 1-2 备份

产出：`02-API服务.md` —— 供应商对照表 + 推荐 + 风险点
```

### 4C：硬件型（IoT/传感器/算力）

```markdown
## 调研主题 2：硬件/算力供应

围绕 <项目定义>，<关键资源> 怎么搞？

1. BOM 盘点：核心硬件清单 + 至少 3 个供应商对比
2. 每个 SKU 的事实：
   - 单价 / MOQ
   - 交付周期
   - 兼容性（接口 / 协议）
   - 供应链稳定性
   - 售后保障
3. 个人开发者主流选择：DIY 圈最常用什么？为什么？
4. 推荐方案：主选 + 备份 + 升级路径

产出：`02-硬件供应.md` —— BOM 表 + 供应商对比 + 风险点
```

## 主题 3：开源生态

```markdown
## 调研主题 3：开源项目

围绕 <项目定义>，GitHub / Gitee 上做同类事情的开源项目至少找 10 个。

1. 可参考项目盘点：至少 10 个
   关键词参考：[根据项目领域填 3-5 个英文+中文关键词]
2. 每个项目的事实：
   - Star / 最近 commit / License
   - 用什么技术栈
   - 做了什么（核心能力）
   - 没做什么（短板）
   - 我能复用什么模块
   - 集成难度（直接 fork / 拆模块 / 仅参考）
3. 生态拼图：哪个项目擅长哪一块
4. 推荐组合：基于 <开源意图>，应该 fork / 借鉴哪几个？

产出：`03-开源项目.md` —— 项目清单 + 复用矩阵 + 初版代码源建议
```

**填充技巧**：
- 关键词参考要给 3-5 个，包含中英文，避免 sub-agent 只搜英文漏掉国内项目
- 如果 `<开源意图>` 是 A（从 0 自己写），把第 4 问改为"了解生态即可，无需选 fork 对象"

## 主题 4：实现方案

```markdown
## 调研主题 4：实现方案

把系统拆成以下模块，每个模块给业内主流方案 + 推荐：

| 模块 | 调研维度 |
|------|---------|
| <模块 1> | [根据模块名展开 2-3 个调研维度] |
| <模块 2> | ... |
| <模块 3> | ... |
| <模块 4> | ... |
| <模块 5> | ... |
| <模块 6> | ... |

每个模块产出：
- 业内主流 2-3 个方案 + 各自优劣
- 推荐用什么 + 理由（要写"为什么不用 X"）
- 大致工作量预估

产出：`04-实现方案.md` —— 方案对比 + 推荐技术栈 + 架构图（Mermaid）
```

**填充技巧**：常见模块的调研维度参考：

| 模块名 | 推荐调研维度 |
|--------|------------|
| 数据/资源接入 | 接收 + 清洗 + 存储的主流方案 |
| 存储层 | 历史数据存哪 / 实时缓存 / 用户数据 |
| 业务逻辑 | 核心算法 / 规则引擎 / 任务调度 |
| 前端 UI | Web 框架 / 组件库 / 关键图表/可视化 / 实时通信 |
| 推送通道 | 飞书 / 微信 / 钉钉 / Telegram / 邮件，接入难度+成本 |
| 部署运维 | 本地 / 云服务器 / Serverless，每月预算估算 |
| 认证/权限 | OAuth / JWT / RBAC，开箱方案 |
| 计费/订阅 | Stripe / 支付宝 / 微信支付 / Paddle 对比 |

## 最终决策汇总

```markdown
## 最终决策汇总：`05-决策汇总.md`

1. 产品形态决策：长什么样（首页结构 + 功能形态）
2. 关键资源决策：用什么 + 切换备份
3. 开源复用决策：fork / 借鉴哪几个项目（如果开源意图=A 则跳过）
4. 技术栈决策：6 模块各自选什么
5. 架构总图：一张端到端架构图（Mermaid）
6. 成本估算：每月预算
7. 风险：最该警惕的坑
```

## 主题适用性判断

并不是所有项目都需要 4 个主题。判断规则：

| 主题 | 跳过场景 | 替换建议 |
|------|---------|---------|
| 主题 1 产品形态 | 内部工具且无任何参照系 | 改成"用户场景调研"（访谈 3 个目标用户） |
| 主题 2 关键资源 | 项目完全不依赖外部资源 | 改成"核心算法调研" |
| 主题 3 开源生态 | 开源意图=A 且高度自定义 | 缩小为"了解生态"，与主题 4 合并 |
| 主题 4 实现方案 | 永远不跳过 | — |

**重要**：跳过主题前必须问用户确认，并在生成的提示词中说明跳过原因。
