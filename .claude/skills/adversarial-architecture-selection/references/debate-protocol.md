# 3 阶段辩论协议详细规范

> 详细规范 5 角色 Agent Team 的 Phase 0 → 1 → 2 → 3 完整流程。Lead 执行时按本规范操作。

---

## Phase 0：候选项目分配

### 输入
- `specs/research/03-开源项目.md`（或用户直接提供的候选清单）
- 项目定位（一句话）

### 步骤

#### 0.1 确定候选数 N
读取候选清单，数清楚 N。

#### 0.2 按规则分配

| N | 分配方案 | 团队配置 |
|---|---------|---------|
| **0** | 无候选——终止流程，告知用户 | — |
| **1** | 单候选——无需对抗，直接进入 Phase 3 简化版（Lead 自行评估）| — |
| **2** | 代言人 1 owns 项目 A；代言人 2 owns 项目 B；**代言人 3 改为"备用红队"**（强化反方）| 2 代言人 + 2 红队 + 1 集成评估师 |
| **3** | 1 代言人 = 1 项目（理想态）| 3 代言人 + 1 红队 + 1 集成评估师 |
| **4** | 1 代言人 owns 2 项目，其余各 1 | 同上 |
| **5** | 2/2/1 分配 | 同上 |
| **6** | 2/2/2 分配 | 同上 |
| **7** | 3/2/2 分配 | 同上 |
| **8+** | 均匀分配，但 Lead 应警告"候选数过多，建议先粗筛"| 同上 |

#### 0.3 分配原则
- 技术栈/定位**相近**的项目 → 合并给同一代言人（便于同一视角对比）
- 技术栈/定位**差异大**的项目 → 分给不同代言人（保持辩论多样性）

举例：
- 5 个候选都是 Python 量化框架 → 按"是否支持实时数据"分组
- 候选混合 Python + JavaScript → 按语言分组

#### 0.4 写入分配文档
路径：`specs/research/debate/00-任务分配.md`

格式：
```markdown
# 候选项目分配（Phase 0）

总候选数 N = {N}
项目定位：{PROJECT_DEFINITION}

## 分配结果

### 代言人 1 owns
- 项目 X（{URL}）
- 项目 Y（{URL}）（如有）
理由：{为什么把这些合并}

### 代言人 2 owns
- 项目 Z（{URL}）
理由：{为什么单独}

### 代言人 3 owns
- 项目 W（{URL}）
理由：{为什么单独}

### 红队
固定 1 个（或 N=2 时 2 个）

### 集成评估师
固定 1 个

## 总体策略
{这次对抗调研的总体策略，如"我们关注的是数据接入层的差异，所以..."}
```

## Phase 1：独立深挖

### 输入
- Phase 0 的 `00-任务分配.md`
- 各候选项目的源码或文档

### 执行规则

#### 1.1 并行启动
所有 5 个 teammate（N=2 时是 5 个 = 2 代言人 + 2 红队 + 1 集成评估师）**同时**启动。Lead 不指定启动顺序。

#### 1.2 严禁互相通信
Phase 1 期间任何 teammate 之间**禁止** mailbox 通信。原因：
- 防止"早期共识偏见"——某个 paper 先出来会被后写的 paper 引用，丧失独立性
- 强制每份 paper 是独立完整的论证

如果 Lead 检测到 teammate 之间通信，立即叫停。

#### 1.3 时间预算
- 单项目代言人：≤ 30 分钟
- 多项目代言人（owns 2+ 项目）：每多 1 项目 +20 分钟，上限 90 分钟
- 红队：≤ 30 分钟（不需深读源码，skim 即可）
- 集成评估师：≤ 30 分钟

#### 1.4 完成标志
每个 teammate 写完产物后 idle。Lead 监控所有 teammate 的状态：
- 所有 teammate idle → 进入 Phase 2
- 某个 teammate 持续 active 超时（如 > 90 分钟）→ Lead 介入检查（不催促，问是否遇阻）

### 输出
```
specs/research/debate/
├── position-paper-{project-1}.md       # 代言人 1 owns 项目 1
├── position-paper-{project-2}.md       # 代言人 1 owns 项目 2（如有）
├── position-paper-{project-3}.md       # 代言人 2 owns 项目 3
├── position-paper-{project-4}.md       # 代言人 3 owns 项目 4
├── red-team-position.md                # 红队
└── integration-assessment.md           # 集成评估师
```

## Phase 2：法庭辩论

### 输入
Phase 1 产出的所有文档。

### 执行规则

#### 2.1 启动前准备
Lead 必须做的事：
1. 检查所有 paper 是否包含第 6 维"致命缺陷自述"——缺失则要求 teammate 补全才能进 Phase 2
2. 把所有 N 份 position paper **随机打乱顺序**——记录打乱后的顺序到内部 log
3. 通过 mailbox 把打乱后的 paper 列表广播给所有 teammate

为什么要打乱？防位置偏见——LLM 评估时第一份 paper 天然占优势。

#### 2.2 第一轮辩论

**步骤 A：红队发起质疑**
- 对**每份 position paper** 单独发 3 条质疑
- 总质疑数 = 3N（N 是 paper 总数，不是代言人数）
- 每条质疑必须带证据（源码 line / commit / issue / 数据）
- 通过 mailbox 发送给对应代言人

**步骤 B：集成评估师挑战单 fork 派**
- 对**所有 position paper** 各发 1 条复合挑战
- 内容：基于 integration-assessment 中的评分，质问"为什么不选复合方案 X"
- 通过 mailbox 发送

**步骤 C：代言人回应**
- 对自己 owns 项目的每份 paper 收到的每条质疑**逐条**回应
- 每条回应 ≤ 200 字
- 可承认 / 反驳 / 提出反质疑
- ⚠️ 禁止跳过任何一条
- ⚠️ 禁止主动放弃任一份 paper

#### 2.3 共识判断
所有代言人回应完毕后，Lead 判断：

| 状态 | 处理 |
|------|------|
| 出现明显占优方案（多数 paper 被驳倒）| 进入 Phase 3 |
| 出现明显否决方案（红队论点全胜）| 进入 Phase 3，决策可能是"全自建"|
| 仍胶着、无明显赢家 | 启动第 2 轮 |

#### 2.4 第二轮辩论（如需要）

第 2 轮只针对第 1 轮**未消解的争议点**：
- Lead 列出"剩余争议点清单"（≤ 5 个）
- 相关 teammate 针对争议点再发起 1 条质疑/回应
- 第 2 轮完毕后**强制**进入 Phase 3，无第 3 轮

为什么限 2 轮？业界研究表明 MAD（Multi-Agent Debate）的边际收益在 2-3 轮后急剧下降，且 token 成本飙升。

#### 2.5 反偏见硬约束（执行时确认）
- ⚠️ Lead **禁止在 Phase 2 中表态**——只观察、只协调，不发表对哪个 paper 更优的意见
- ⚠️ 每条质疑回应 ≤ 200 字（防长度偏见）
- ⚠️ 不允许同一 teammate 在同一轮多次回复对同一质疑
- ⚠️ 不允许 teammate 在 Phase 2 修改自己的 Phase 1 paper（如要修改，必须公开声明"我撤回 paper 第 X 节"）

### 输出
`specs/research/debate/debate-transcript.md`：完整辩论记录，按时间顺序排列，便于 Phase 3 引用。

## Phase 3：Lead 综合判决

### 输入
- 所有 position papers（N 份）
- red-team-position.md
- integration-assessment.md
- debate-transcript.md

### 执行规则

#### 3.1 Lead 单独工作
Phase 3 **不调用任何 teammate**。Lead 自己读完所有文档、综合权衡、写决策。

#### 3.2 判决方法论

按"幸存论点"加权：
- 一条论点被对方驳倒 → 失效（不计入 Lead 的判决依据）
- 一条论点被对方质疑但代言人成功回应 → 幸存，权重正常
- 一条论点未被任何对方质疑 → 幸存，但 Lead 要警惕"是否真的没人质疑"（可能是被忽视）

判决时不允许的依据：
- ❌ "代言人 X 论证更长 / 更详细"（长度偏见）
- ❌ "代言人 X 的论点先出现"（位置偏见）
- ❌ "代言人 X 用词更专业"（措辞偏见）
- ❌ "我个人偏好项目 Y"（self-preference 偏见）

允许的依据：
- ✅ 7 维度的可量化数据（改造成本、缺陷数、扩展点数）
- ✅ 对方质疑后无回应或回应空洞
- ✅ 集成评估师的客观评分
- ✅ 多个独立来源指向同一结论

## 团队清理

Phase 3 写完后：

```
1. Lead 通过 mailbox 通知所有 teammate "辩论结束，准备 shutdown"
2. 逐个 shutdown teammate（按 spawn 顺序的反序）
3. 检查无 teammate active 后，清理 team 资源
4. 通知用户对抗调研完成，下一步进入 brainstorming
```

---

## 异常处理

| 异常 | 处理 |
|------|------|
| teammate Phase 1 超时（> 90 min）| Lead 介入询问，不催促，问是否遇阻 |
| 某 paper 缺第 6 维（致命缺陷自述）| 退回让代言人补全，不进 Phase 2 |
| 红队"和稀泥"（说"X 项目其实也行"）| Lead 介入纠正，要求红队回归全否定立场 |
| 代言人在 Phase 2 想倒戈支持别人 | 警告——代言人立场固定，倒戈意愿可在 Phase 3 由 Lead 综合时参考 |
| 第 2 轮后仍胶着 | 强制进 Phase 3，由 Lead 凭幸存论点判决 |
| Lead 自己拿不准 | 写入 `06-架构基线决策.md` 的 "Open Questions" 一节，留给 brainstorming 阶段处理 |
