# PRD 反模式清单

> 写完 PRD 必须过一遍这份清单。出现任何一条都要回去修改。

---

## 致命反模式（看到立刻打回）

### ❌ 1. PRD 没有 Why（缺背景与目标）
**症状**：直接列功能，没有"为什么做这个"
**为什么是问题**：团队不知道做这事的优先级判断依据，做歪了不知道
**修复**：补 §2 项目背景与目标

### ❌ 2. 没有 Out-of-Scope（范围会爆炸）
**症状**：只写"做什么"，不写"不做什么"
**为什么是问题**：开发到一半会被加需求，原范围保不住
**修复**：每个 Scope 至少配 3-5 条 Out-of-Scope

### ❌ 3. 验收标准写"系统应该好用"
**症状**：验收标准模糊（"友好"/"高效"/"快速"）
**为什么是问题**：无法机器验证，QA 没法测试，AI 没法理解
**修复**：用 Given/When/Then 格式 + 具体数字

### ❌ 4. 把技术栈写进 PRD
**症状**：「使用 React + PostgreSQL + Redis」
**为什么是问题**：剥夺工程师的最优解空间，污染文档边界
**修复**：搬到 TRD（技术方案文档）

### ❌ 5. 没有 Open Questions（假装一切清楚）
**症状**：PRD 看上去"完美"，没有任何不确定项
**为什么是问题**：项目里没有任何东西是"全部清楚"的，假装清楚等于挖坑
**修复**：明确列出 3-10 个待决策的问题

### ❌ 6. MVP 范围 > 8 个功能
**症状**：MVP 列了 15 个 P0 功能
**为什么是问题**：这不是 MVP，是完整产品。会做不完
**修复**：砍到 5 个以内，其他移到 v1.1+

## 严重反模式（必须修复）

### ⚠️ 7. 优先级全是 P0
**症状**："这些都很重要"
**修复**：用 RICE / MoSCoW 强制分级，P0 不超过 5 个

### ⚠️ 8. 用户画像写"所有用户"
**症状**：「目标用户：所有想做量化的人」
**修复**：写具体画像 + 反画像

### ⚠️ 9. 用户故事缺 Who 或 Why
**症状**：「系统支持自选股管理」
**修复**：改成「作为 X 用户，我希望 Y，以便 Z」

### ⚠️ 10. 没有详细功能描述
**症状**：功能列表只有名字，没有输入/输出/流程
**修复**：每个功能补"触发/输入/流程/输出/边界条件"

### ⚠️ 11. 缺非功能需求
**症状**：性能 / 安全 / 合规一个字没提
**修复**：单独写 §7，至少覆盖性能、可用性、安全

### ⚠️ 12. 把 NFR 埋在功能段落里
**症状**：性能要求散落在各个功能下
**修复**：抽出来统一放 §7

### ⚠️ 13. 没有度量指标
**症状**：「上线后看反馈」
**修复**：明确北极星指标 + AARRR 关键数字

### ⚠️ 14. 把里程碑写得太死
**症状**：日期精确到天，没有 buffer
**修复**：留 20% buffer，明确风险缓冲

## 一般反模式（建议修复）

### 🟡 15. 文档信息缺失
**症状**：没有版本号 / 状态 / 作者 / 变更记录
**修复**：补 §1，每次重大改动更新版本

### 🟡 16. 版本号永远 v0.1
**症状**：从立项到上线都是 v0.1
**修复**：MVP 上线版本 = v1.0，重大变更 = v1.1 / v2.0

### 🟡 17. 状态永远 Draft
**症状**：PRD 已上线但状态还是 Draft
**修复**：评审通过改 Approved，废弃改 Deprecated

### 🟡 18. 没有反指标
**症状**：度量指标全是正向指标
**修复**：加反指标防止"指标作弊"（如"留存高但深度使用率低"）

### 🟡 19. 风险写得太抽象
**症状**：「可能有合规风险」
**修复**：具体到 What/Severity/Probability/Mitigation

### 🟡 20. 没有依赖清单
**症状**：上线前才发现卡了第三方
**修复**：列依赖项 + 期望就绪时间 + 风险等级

## AI 时代特有反模式

### 🤖 21. PRD 写得太"散"，AI 无法消费
**症状**：内容散落在各处，缺乏结构化字段
**修复**：用标准 14 章结构，关键字段用统一格式

### 🤖 22. 验收标准无法机器验证
**症状**：「系统应该聪明地处理用户输入」
**修复**：Given/When/Then 格式，每条都可写自动化测试

### 🤖 23. 没有"Alternatives Considered"
**症状**：只写最终方案，不写考虑过的备选
**修复**：把 Brainstorming 中被否的方案 + 原因写进 PRD
（这是 Spec-kit 的核心建议）

### 🤖 24. 边界条件不完整
**症状**：只写正常路径，不写异常 / 边界
**修复**：每个功能至少写 3-5 个边界条件

## 反模式速查表

| 类型 | 反模式 | 自检方法 |
|------|--------|---------|
| 致命 | 缺 Why | 翻到 §2，看有没有"为什么是现在" |
| 致命 | 缺 Out-of-Scope | 翻到 §5，看有没有 ❌ 标记 |
| 致命 | 验收太抽象 | 翻到 §9，能否写自动化测试？ |
| 致命 | 含技术栈 | 全文搜 React/Vue/Postgres 等 |
| 致命 | 假装无 OQ | 翻到 §13，至少 3 个？ |
| 致命 | MVP > 8 个 | 翻到 §10，数 P0 |
| 严重 | 全 P0 | 翻到 §10，统计 P0 比例 |
| 严重 | 全员目标用户 | 翻到 §3，有反画像？ |
| 严重 | 故事缺 Who/Why | 翻到 §4，每条都有"作为...以便..." |
| 严重 | 缺详细描述 | 翻到 §6，有"触发/输入/输出/边界"？ |
| 严重 | 缺 NFR | 翻到 §7，覆盖性能/安全/合规？ |
| 严重 | 缺度量 | 翻到 §11，有北极星 + AARRR？ |
| AI | 缺备选记录 | 有"Alternatives Considered"？ |
| AI | 边界不全 | 每功能 ≥ 3 个边界条件？ |

## 好 PRD 5 大特征（正反对照）

1. ✅ **目的清晰**：3 秒能说出这产品的核心价值
2. ✅ **范围明确**：MVP 不超过 5 个 P0，Out-of-Scope 完整
3. ✅ **可验证**：每个功能都有 Given/When/Then 验收
4. ✅ **不越界**：只写 What/Why，不写 How
5. ✅ **诚实**：标明 Open Questions / 风险 / 决策记录

---

## 写完 PRD 必做的 3 件事

1. **过反模式清单**（本文档）
2. **找一个不在项目里的人读一遍**——能在 5 分钟内说清楚产品是什么，PRD 就合格
3. **存档当前版本**（git commit），后续修改保留可追溯

---

**记住**：PRD 是契约不是说明书。契约的核心是"边界 + 验收"。一份没有边界 / 没有验收的 PRD 等于没有契约。
