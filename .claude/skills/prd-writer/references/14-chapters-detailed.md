## 第 1 章 · 文档信息（必备 ★）

**目的**：让读者一眼知道"我是不是在看最新版"。

**模板片段**：

```markdown
## 1. 文档信息

| 项 | 值 |
|---|---|
| 文档名称 | <项目名> · 产品需求文档 |
| 版本 | v0.3 |
| 状态 | Draft / Review / Approved / Deprecated |
| 作者 | <PM 姓名> |
| 创建日期 | YYYY-MM-DD |
| 最后更新 | YYYY-MM-DD |
| 评审人 | <跨职能评审清单> |

### 变更记录
- v0.3 (YYYY-MM-DD) 增加风控模块章节
- v0.2 (YYYY-MM-DD) 补充非功能需求
- v0.1 (YYYY-MM-DD) 初稿
```

**反模式**：没有版本号 · 变更记录写"小修改" · 状态永远是 Draft

## 第 2 章 · 项目背景与目标（必备 ★）

**目的**：回答"为什么这件事现在值得做"。

**模板片段**：

```markdown
## 2. 项目背景与目标

### 2.1 背景
<2-3 段，说清楚：
- 当前的世界是什么样？
- 用户当前怎么解决这个问题？
- 现状有什么不够的地方？>

### 2.2 目标（Goals）
- **业务目标**：<6 个月内 X 用户 / 留存 Y / 收入 Z>
- **产品目标**：<用户完成某事的时长从 X 缩短到 Y>
- **学习目标**（如果是探索性项目）：<我们希望验证什么假设>

### 2.3 非目标（Non-Goals / 不做什么）
- 不做 <某事>，因为 <理由>
- 不做 <另一事>，留到 v2

### 2.4 战略对齐
本项目对齐 <公司年度战略 / OKR / 产品矩阵>
```

**反模式**：写成 KPI 罗列 · 没有"为什么是现在" · 缺 Non-Goals

## 第 3 章 · 目标用户与画像（必备 ★）

**目的**：让所有团队成员脑里有同一个用户形象。

**模板片段**：

```markdown
## 3. 目标用户与画像

### 3.1 主要画像 P1：<画像名称>
- **背景**：<职业、年龄段、技术水平>
- **目标**：<他来用我们产品要达成什么>
- **痛点**：<现在遇到的具体困难，3-5 条>
- **使用频次**：<每天/每周多少次>
- **设备**：<PC / 手机 / Pad / 多设备>
- **数字代表**：<他每月愿意花多少钱 / 投入多少时间>

### 3.2 次要画像 P2：<画像名称>
（同上结构）

### 3.3 反画像（明确不服务的人）
- <某类用户>，原因：<...>
- <另一类用户>，原因：<...>
```

**反模式**：写"所有想做 XX 的人"（=没写）· 只有人口学没有行为学

## 第 4 章 · 用户故事 / 使用场景（必备 ★）

**目的**：把抽象目标变成具体场景，让团队产生共情。

**模板片段**（INVEST 格式）：

```markdown
## 4. 用户故事

### US-01 <故事标题>
作为 <某类用户>
我希望 <做某事> <在某情境下>
以便 <达成某目的>

**触发情境**：<用户在什么时间、地点、心情下会用>
**期望结果**：<用户完成后是什么状态>

### US-02 ...
```

**INVEST 自检**：
- **I**ndependent 独立可交付
- **N**egotiable 可协商
- **V**aluable 有用户价值
- **E**stimable 可估算
- **S**mall 足够小
- **T**estable 可测试

**反模式**：写成功能列表（"系统支持 XX"）· 丢失 Who/Why · 一个故事跨越多个场景

## 第 5 章 · 功能列表与范围（必备 ★）

**目的**：一句话告诉团队"这版到底要造哪些东西"。

**模板片段**：

```markdown
## 5. 功能列表与范围

### 5.1 In-Scope（MVP v1.0 必须有）
| ID | 功能名 | 一句话描述 | 关联用户故事 | 优先级 |
|---|---|---|---|:--:|
| F1 | 自选股 Dashboard | 实时展示用户关注的股票 | US-01, US-03 | P0 |
| F2 | AI 智能选股 | 自然语言转选股查询 | US-02 | P0 |
| F3 | 早盘简报 | 每天 8:00 推送 | US-04 | P1 |
| F4 | 异动预警 | 涨跌 >5% 立刻推送 | US-05 | P1 |

### 5.2 Out-of-Scope（v1.0 不做）
- ❌ 自定义策略回测 → 移到 v1.1
- ❌ 多账户管理 → 移到 v2.0
- ❌ 移动端 APP → 暂不考虑
- ❌ 国际化（英文版）→ 暂不考虑

### 5.3 优先级定义
- P0 = 必须有（缺了产品不能上线）
- P1 = 应该有（缺了用户体验大打折扣）
- P2 = 可以有（锦上添花）
```

**反模式**：MVP 范围 > 8 个功能 · 没有 Out-of-Scope · 优先级全是 P0

## 第 6 章 · 详细功能描述（必备 ★）

**目的**：每个功能的输入/输出/流程/边界都明确，让工程师 / AI 不需要"猜"。

**模板片段**（推荐"卡片式"，一个功能一个 4 元组）：

```markdown
## 6. 详细功能描述

### F1 自选股 Dashboard

**触发**：<什么动作触发这个功能 — 用户点击 / 定时器 / API 调用>

**输入**：
- `param_name`: <类型，含义，约束>（必填/选填，默认值）
- ...

**核心流程**：
1. <步骤 1>
2. <步骤 2>
3. <...>

**输出**：
- `field_name`: <类型，含义>
- ...

**边界条件**：
- <异常情况 1> → <处理方式>
- <异常情况 2> → <处理方式>

**异常处理**：
- <错误类型> → <错误码 / 用户提示>

**性能要求**：<具体数字，如 p95 < 200ms>

**业务规则**：
- <规则 1>
- <规则 2>
```

**反模式**：
- ❌ 只写功能名（"支持自选股"）不写流程
- ❌ 没有边界条件 / 异常处理
- ❌ 把实现细节（"用 Redis 缓存"）写进来
- ❌ 性能要求写"应该比较快"

## 第 7 章 · 非功能需求（必备 ★）

**目的**：质量底线。NFR 必须单独成节，不能埋在功能段落里。

**模板片段**：

```markdown
## 7. 非功能需求（NFR）

### 7.1 性能
- 首屏加载 p95 < 2 秒
- API 响应 p95 < 500ms
- 大列表渲染（>1000 条）< 1 秒
- 实时数据延迟 < 100ms

### 7.2 可用性
- 系统每月可用时间 ≥ 99.5%（即每月停机 ≤ 3.6 小时）
- 关键业务时段（A 股 9:15-15:00）可用性 ≥ 99.95%
- 故障恢复 RTO < 5 分钟，RPO < 1 分钟

### 7.3 安全
- 用户密码加密存储（bcrypt / argon2）
- API Key 加密存储（AES-256）
- 实盘下单前必须 2FA 二次确认
- 审计日志保留 7 年（金融场景）
- 不存储用户身份证号 / 银行卡号

### 7.4 合规
- 符合 <相关法规，如证监会程序化交易细则 / GDPR / 个保法>
- 不提供 <红线行为，如代客理财>

### 7.5 数据
- 历史数据完整性：缺失率 < 0.1%
- 数据备份频率：每天 1 次
- 数据保留：原始数据 ≥ 5 年

### 7.6 可访问性（A11y）
- 支持键盘导航
- 颜色对比度 ≥ WCAG AA 标准
- 支持屏幕阅读器

### 7.7 国际化（i18n）
- v1.0 仅中文（简体）
- 字符编码 UTF-8
```

**反模式**：NFR 写"系统应该快" · 没有具体数字 · 缺合规这一块

## 第 8 章 · UI / 交互说明（推荐 ☆）

**目的**：让设计 / 前端理解"要呈现什么 + 怎么交互"。

**模板片段**：

```markdown
## 8. UI / 交互说明

### 8.1 信息架构
（一张 Mermaid 图，展示主导航 / 子页面层级）

### 8.2 关键页面
- **首页 / Dashboard**：<关键信息卡片 + 行为按钮>
- **选股页**：<输入框 + 结果列表>
- **设置页**：<...>

### 8.3 关键交互流程
- 选股：用户输入 → 加载状态 → 结果展示 → 加入自选
- 异动接收：推送通知 → 点击 → 进入详情

### 8.4 设计稿链接
- Figma: <link>
- 关键 Frame 命名：<对齐通用语言>
```

**反模式**：把 PRD 写成设计稿评审 · 过度详细配色 / 字号（那是设计稿的事）

## 第 9 章 · 验收标准（必备 ★）

**目的**：怎么算"做完了"——必须可执行 / 可机器验证。

**模板片段**（Given/When/Then 格式）：

```markdown
## 9. 验收标准

### AC-F1-01：自选股添加成功
**Given** 用户已登录且自选股数 < 上限（30）
**When** 用户在搜索框输入"贵州茅台"并点击"加入自选"
**Then** 该股票出现在自选股列表，且实时价格在 1 秒内展示

### AC-F1-02：自选股达到上限
**Given** 用户的自选股已有 30 只
**When** 用户尝试添加新股票
**Then** 系统提示"自选股已达上限，请先移除其他股票"，且操作不执行

### AC-F2-01：自然语言选股 3 秒内出结果
**Given** 系统状态正常
**When** 用户输入"最近 3 天放量上涨的新能源股"
**Then** 3 秒内返回符合条件的股票列表（至少 1 只）

### AC-NFR-01：性能验收
**Given** 1000 用户并发
**When** 用户访问 Dashboard
**Then** p95 加载时间 < 2 秒
```

**反模式**：写"功能跑通即可" · 没有量化条件 · 缺异常路径验收

## 第 10 章 · 优先级 / MVP 范围（必备 ★）

**目的**：明确"先做哪一刀"，防止范围爆炸。

**模板片段**（RICE / MoSCoW 框架）：

```markdown
## 10. 优先级 / MVP 范围

### 10.1 MVP 范围（v1.0）
**核心 5 件事**（其他都暂缓）：
1. F1 自选股 Dashboard
2. F2 AI 智能选股
3. F3 早盘简报
4. F4 异动预警
5. F7 飞书推送

### 10.2 MVP 验证假设
通过 MVP 我们要验证：
- 假设 1：用户愿意把同花顺替换成我们的工具
- 假设 2：自然语言选股能解决 80% 的用户场景

### 10.3 v1.1 计划（MVP 之后）
- F5 策略回测
- F6 多账户管理

### 10.4 优先级评估方法
采用 RICE 评分：Reach × Impact × Confidence / Effort
| 功能 | R | I | C | E | RICE |
|---|---|---|---|---|---|
| F1 | 100 | 3 | 0.9 | 2 | 135 |
```

**反模式**：MVP 是"完整产品" · 没有验证假设 · 没拆 v1.1 / v2.0

## 第 11 章 · 度量指标（必备 ★）

**目的**：怎么算"成功"——上线后我们看什么数据判断。

**模板片段**（North Star + AARRR）：

```markdown
## 11. 度量指标

### 11.1 北极星指标（North Star）
**用户每周主动检查股票的次数**（DAU/WAU 不够，要看深度使用）

### 11.2 关键指标
**Acquisition（拉新）**
- 月新增用户 ≥ X
- 渠道转化率 ≥ Y%

**Activation（激活）**
- 注册后 24h 内添加 ≥ 3 只自选股的用户占比 ≥ 60%

**Retention（留存）**
- 次日留存 ≥ 40%
- 7 日留存 ≥ 25%
- 30 日留存 ≥ 15%

**Revenue / Referral**（如适用）

**Referral（推荐）**
- 用户主动分享 N 次

### 11.3 反指标（防止"指标作弊"）
- 自选股数量平均 ≥ 5（防止只为完成新手任务）
- 推送打开率 ≥ 20%（防止纯垃圾推送拉留存）

### 11.4 度量频率
- 日报：DAU / 关键操作转化
- 周报：留存 / 北极星
- 月报：完整 AARRR
```

**反模式**：只看 DAU · 没有反指标 · 没说怎么收集

## 第 12 章 · 依赖与约束（推荐 ☆）

**目的**：标出"卡在哪 / 等谁"。

**模板片段**：

```markdown
## 12. 依赖与约束

### 12.1 外部依赖
| 依赖项 | 提供方 | 期望就绪时间 | 风险等级 |
|---|---|---|:--:|
| AKShare 数据接入 | 第三方开源 | 已就绪 | 低 |
| 飞书机器人接入 | 飞书开放平台 | 1 周内 | 中 |
| 阿里云 ECS | 阿里云 | 3 天内 | 低 |

### 12.2 内部依赖
- 设计稿 by 设计团队（截止 X 日）
- 数据 API by 数据团队（截止 Y 日）

### 12.3 约束
- 预算：每月服务器费用 ≤ ¥500
- 时间：6 周内上线 MVP
- 团队：1 人独立完成（自己 + AI 编程辅助）
- 法律：A 股数据合规、不能商业化（v1.0 个人自用）
```

**反模式**：缺这一节（上线时才发现卡了）· 没有就绪时间 / 风险等级

## 第 13 章 · 风险与对策 / 开放问题（必备 ★）

**目的**：标出"还不确定什么 + 已知风险怎么对冲"。

**模板片段**：

```markdown
## 13. 风险与开放问题

### 13.1 风险登记
| ID | 风险 | 严重 | 概率 | 对策 |
|---|---|:--:|:--:|---|
| R-01 | AKShare 接口变更 / 限频 | 高 | 中 | 准备 BaoStock 作为备用源 |
| R-02 | 推送过频导致用户骚扰 | 中 | 高 | 设置每日推送上限 + 用户可关闭 |
| R-03 | 涉及"投资建议"红线 | 高 | 低 | 全程加免责声明 |

### 13.2 开放问题（Open Questions）
- [ ] OQ-01：飞书机器人能否突破免费版每天 100 条限制？@PM 跟进
- [ ] OQ-02：是否需要支持港股？决策点 v1.1 之前
- [ ] OQ-03：用户密码是否需要复杂度要求？@设计师 决策
- [ ] OQ-04：异动预警的"重大新闻"如何定义？@PM 召集脑暴

### 13.3 决策记录
（决策一旦做出，把对应的 OQ 标记为已闭环）
- ✅ DR-01：v1.0 不做美股 → 见 ADR-001
```

**反模式**：假装一切清楚（没有 Open Questions）· 风险写"可能有问题"（不具体）

## 第 14 章 · 里程碑 / 时间计划（推荐 ☆）

**目的**：什么时候做完。

**模板片段**：

```markdown
## 14. 里程碑

### 14.1 关键节点
| 里程碑 | 目标日期 | 交付物 | 责任人 |
|---|---|---|---|
| M1 PRD Approved | YYYY-MM-DD | 本文档 v1.0 | PM |
| M2 技术方案完成 | YYYY-MM-DD | TRD + 架构图 | 工程师 |
| M3 设计稿完成 | YYYY-MM-DD | Figma | 设计 |
| M4 开发完成 | YYYY-MM-DD | 可演示版本 | 工程师 |
| M5 测试完成 | YYYY-MM-DD | 测试报告 | QA |
| M6 上线 | YYYY-MM-DD | 生产环境 | 工程师 |

### 14.2 6 周节奏
- 第 1 周：技术方案 + Figma
- 第 2-3 周：前端 + 后端并行开发（基于契约）
- 第 4 周：联调 + 修 Bug
- 第 5 周：测试 + 灰度
- 第 6 周：上线 + 观察期

### 14.3 风险缓冲
预留 1 周 buffer，应对意外延期
```

**反模式**：时间表写得太死（没 buffer） · 没说责任人 · 里程碑没交付物

---

## 章节使用建议

### 完整版（适用于：中大型项目 / 跨团队 / B 端 SaaS）
全部 14 章

### 精简版（适用于：个人项目 / 小团队 / 探索性 MVP）
保留 #1, #2, #3, #5, #6, #9, #10, #11, #13（共 9 章）

### 极简版（适用于：1 人项目 / 1 周内交付）
保留 #2, #5, #6, #9, #13（共 5 章）

---

**记住**：PRD 不是越长越好。长到说清楚为止，不再多写一字。
