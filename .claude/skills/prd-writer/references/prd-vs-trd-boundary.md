# PRD vs TRD 边界判定指南

> 核心问题：哪些写进 PRD，哪些必须留给 TRD（技术方案文档）？

---

## 一刀切判断法

> **工程师能换一种实现仍满足该条 → 留在 PRD**
> **工程师不能换实现 → 那是技术决定，搬去 TRD**

---

## 边界对照表

| 内容类型 | 进 PRD ✅ | 进 TRD ❌ |
|---------|----------|----------|
| **性能** | 「首屏 ≤ 2s」 | 「用 Redis 缓存 + CDN」 |
| **功能行为** | 「自然语言转选股 3 秒内出结果」 | 「用 Claude API + Function Calling」 |
| **数据要求** | 「保留历史行情 ≥ 5 年」 | 「用 TimescaleDB 分表存储」 |
| **接口契约** | 「需要异动推送接口，参数：股票/阈值」 | 「OpenAPI YAML + 端点路径」 |
| **业务规则** | 「自选股上限 30 只」 | 「用 PostgreSQL CHECK 约束」 |
| **错误处理** | 「数据缺失时返回明确错误码」 | 「错误码用 RFC 7807 格式」 |
| **可用性** | 「9.95% 可用性」 | 「用 Kubernetes 多副本 + 健康检查」 |
| **安全** | 「下单前 2FA」 | 「用 TOTP / SMS 哪种 2FA」 |
| **合规** | 「不能存储身份证号」 | 「用什么加密算法」 |
| **UX** | 「关键操作有 loading 状态」 | 「用 Suspense 还是 useState」 |

---

## 三种立场对照

### 主流立场（反对 PRD 写技术细节）
- **核心论点**：PRD 写 What & Why，TRD 写 How。在 PRD 里指定"用 React"等于剥夺工程师的最优解空间。
- **例外**：可以提"示意性实现"来澄清用例，但不指定具体技术

### 严格立场（PRD 一行技术都不能有）
- **核心论点**：PRD 受众包含市场/客服/法务，写技术细节会污染文档
- **实操**：技术约束、API、数据模型、错误码 → 全部走 TRD

### Spec-Driven / AI 立场（本 Skill 推荐 ⭐）
- **核心论点**：AI 时代 PRD 必须写到"行为可验证"的精度，但**仍不指定技术栈**
- **关键认知**：PRD 不是实现指南，而是生成实现的源——所以要写到行为契约级别
- **实操**：PRD 写行为契约（输入/输出/边界），不写具体类 / 库 / 框架

## 灰色地带处理

某些内容既不像 What 也不像 How，怎么办？

### 灰色 1：数据库 Schema
- ❌ 不写：`CREATE TABLE users (id INT, ...)` —— 这是 TRD
- ✅ 写：「需要存储用户的：基本信息（名称、注册时间）、自选股清单（≤ 30 只）、推送偏好」—— 这是 PRD（数据模型抽象）

### 灰色 2：API 契约
- ❌ 不写：完整 OpenAPI YAML —— 这是 TRD
- ✅ 写：「需要提供异动推送 webhook 接口，包含：股票代码、变动类型、变动幅度、时间戳」—— 这是 PRD（接口抽象）

### 灰色 3：第三方依赖
- ❌ 不写：「集成 Tushare Pro API（密钥见 .env.TUSHARE_TOKEN）」—— 这是 TRD
- ✅ 写：「需要稳定的 A 股行情数据源，支持日级 + 分钟级数据，覆盖至少 10 年历史」—— 这是 PRD（能力要求）
- 🟡 可以写在 PRD 的"约束"章节：「数据源必须满足合规要求，不允许商业化使用受限的数据」

### 灰色 4：性能指标
- ✅ 写：「首屏 p95 < 2s」「API p95 < 200ms」—— 这是 PRD（产品要求）
- ❌ 不写：「用 LCP 指标监控 + Lighthouse 评分 ≥ 90」—— 这是 TRD（实现细节）

## Spec-Driven Development 的 3 条核心规则

适用于"PRD 给 AI Agent 消费"场景：

### 规则 1：不描述实现细节
PRD 是 WHAT 和 WHY，不是 HOW。
- ✅ 用户能在 3 秒内完成搜索
- ❌ 使用 Elasticsearch 实现搜索

### 规则 2：包含足够细节让 AI Agent 能消费
验收标准、边界条件、错误处理都要明确写出来。

### 规则 3：写明 Alternatives Considered
把被否的方案放进 PRD，说明为什么不选——这是 AI 推理时的关键上下文。

## PRD 中"可以提及技术"的 3 个特殊场景

虽然总体原则是 PRD 不写技术，但以下 3 种情况是例外：

### 场景 1：硬性技术约束
当外部约束强制使用某技术时，可以写在 PRD 的"依赖与约束"章节：
- 「必须使用公司统一的 SSO 登录」
- 「必须部署在阿里云（合规要求）」

### 场景 2：技术选型本身是产品差异化点
当某技术选择本身就是产品卖点时：
- 「使用本地大模型，保证数据不出境」
- 「区块链记录交易日志，保证不可篡改」

### 场景 3：受众需要知道技术上下文
当 PRD 的某部分内容涉及合规 / 法律 / 用户教育时：
- 「我们使用 AES-256 加密用户敏感数据」（出于合规说明）
- 「我们不存储用户密码原文」（出于用户信任）

## 自检 Checklist

写完 PRD 后过一遍：

- [ ] 没有出现具体框架名（React/Vue/Django）
- [ ] 没有出现具体数据库名（PostgreSQL/MongoDB），除非是上述 3 个特殊场景
- [ ] 没有出现具体代码示例（除非是 API 契约的抽象描述）
- [ ] 没有指定具体目录结构 / 文件命名
- [ ] 没有指定具体的设计模式（"用 Repository 模式"）
- [ ] 性能要求是"产品级数字"（p95 < 200ms），不是"实现级数字"（QPS / 内存占用）

如果 PRD 里有违反以上的内容，问自己一个问题：
> **"如果工程师用完全不同的技术实现，这条需求还成立吗？"**
>
> - 答 "是" → 留在 PRD
> - 答 "否" → 搬去 TRD
