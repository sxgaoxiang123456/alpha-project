# 测试路由对照表（完整版）

这张对照表把一个 feature 的信号映射到它**所属的场景类别**，再由类别**路由到对应类别的执行器
skill**（见 `tool-mapping.md` 的「类别 → 能力 → 路由」表）去闭环补测。**本 skill 只判类 +
标缺口 + 路由，不写死任何单栈工具**——具体工具由被路由到的执行器 skill 按项目技术栈实例化。
在 feature 收尾分类时读它。表里没有任何项目专有内容——把信号名替换成项目实际使用的叫法，并在
某个信号缺失时优雅降级。

> 方法论（判类 / 缺口标注 / 三层节奏 / 可追溯 / 发布门）统一以 `testing-system-blueprint` skill 为蓝本。

## 类别是一个开放集合

分类的主输出轴是**场景类别**，它**开放、可增减**：下面是当前已识别的类别，但遇到不能干净
归入现有类别的 task，**新增一类并注明**，而不是硬塞进一个不合身的类。判类信号来自
**任务范围标签 + 依赖图 + 契约声明**（再辅以验收标准 AC）。

## 各列含义

- **类别** —— 这部分 task 落在哪个场景类别。决定后续路由到哪个执行器 skill 的是它。
- **判类信号** —— 项目里的什么东西告诉你这部分 task 属于这一类。
- **怎么用** —— 收尾时对这一类该做什么。
- **路由去向** —— 这一类的缺口路由到哪个执行器 skill（具体工具由该 skill 按栈实例化）。

## 对照表

| 类别 | 判类信号 | 怎么用 | 路由去向（执行器 skill） |
|---|---|---|---|
| **单后端** | 一个后端范围的任务标签（如 `[BE]` 风格），或描述只触及后端单元的任务 | 单元测试通常 TDD 阶段已写好；收尾时只**核对覆盖是否存在**，结构性缺口路由出去补 | → `backend-testing`（✅ 执行器已建；按栈解析工具） |
| **单前端** | 一个前端范围的任务标签（如 `[FE]` 风格），或描述只触及前端单元的任务——改动**只落在前端层**（组件 / 页面 / 样式 / 前端路由 / 前端状态），不涉及后端业务逻辑或真后端联调（联调归「完整功能链路」） | 与单后端不同，前端 L0/L1 测试地基常常为零（[FE] 出参常只写"手测"、没装测试运行器）；收尾时核对地基是否存在，并标出视觉回归 / a11y / 跨浏览器 + 响应式 / 契约 mock 缺口路由出去补 | → `frontend-testing`（✅ 执行器已建；按栈接成熟工具 + 配置 + 把视觉契约翻译成断言，唯一人审节点是 L2 视觉基线裁决） |
| **局部前后端** | 改动**同时落在前端和后端**、且这条接缝**不跨出本 feature 边界**（跨多 feature 旅程归「完整功能链路」）；或一个集成范围的标签 / 写着"打通 / 把 X 接到 Y / feature 内端到端"的任务 | 单元全绿之后、宣布 feature 完成之前，补**同一 feature 内** *真前端↔真后端* 的单切片对账——新难点在"起真栈"（环境编排）而非写断言：先用 diff-aware E2E 跑一层黑盒冒烟，再补结构化接缝断言。这是收尾最常见的真实缺口 | → `fullstack-slice-testing`（✅ 执行器已建；按栈起真栈 + 接缝断言；diff-aware E2E 如 gstack `/qa` 非依赖、无则回退） |
| **完整功能链路** | 依赖图拓扑里出现一条 A→B→C 路径、**跨出本 feature 边界横跨多个 feature**，结合该旅程的用户故事 / AC；含非 UI 跳步（定时 / 异步 / 跨通道） | 一旦拓扑显示该旅程**首次变得可达**，点名该链路；它的被测对象（通路）要先被**挖掘**出来（前三格被测对象给定），再按关键性只取一小撮 P0 做**安全网**（非穷举，bug 首现于此=下层漏测、回补下层） | → `full-chain-testing`（✅ 执行器已建；按栈挖通路 + 全系统编排（只 stub 外部第三方）+ 异步/时间/跨通道编排驱动；UI 可走段 diff-aware E2E 如 gstack `/qa` 折叠进此处、非依赖、无则回退） |
| **（可增类…）** | 视项目实际信号而定 | 现有类别装不下时新增并注明判类依据 | → 视该类性质而定（🔧 占位·待建） |

## 推荐可增的候选类：跨模块契约

当本 feature 的输出是另一个模块的输入（A→B），且这个接口有**契约声明**（接口签名 / 对外公开
契约表 / 共享 key / schema）时，这部分 task 可单独归为「跨模块契约」一类。

- **为什么特别推荐它：** 接口接缝是**漂移 / 断裂的高发区**——A 改了输出形状、B 还按旧形状读，
  双方各自的单元测试全绿也照样断；这种悄无声息的 A→B 不匹配，只有契约层能抓到。
- **它是建议而非强制：** 按开放集合处理。项目里有契约声明就把它作为一类点出来；没有契约声明
  时不启用，并在报告里说"信息不足 —— 未找到契约声明"，而不是凭空编一个。

## 怎么用这张表

1. **通篇扫描 + 给任务归类。** 读范围标签（或推断范围），把每个任务落到对应类别。单后端 /
   单前端的工作通常 TDD 循环已覆盖——你的活儿是确认，不是重写。局部前后端（任何在本 feature
   内横跨前端↔后端、或服务↔数据存储的东西）是收尾时最常见的真实缺口。

2. **判定本 feature 整体属于哪一类。** 落在多类时点名主类 + 次类，并简述判类依据。

3. **若启用「跨模块契约」候选类，检查契约。** 对本 feature *暴露* 或 *改动* 的每一个契约，在
   依赖图里找出它的消费者。每个被改动的契约都需要在生产者侧的契约检查，外加对每个消费者做一次
   回归。如果没有契约声明可读，就说"信息不足 —— 未找到契约声明"。

4. **遍历依赖图找完整功能链路。** 这一步撑起了整个 skill 的价值。沿着依赖边追踪，找出任何
   *因为本 feature 刚落地* 而**跨出本 feature 边界、横跨多个 feature**变得完全连通的路径。一条刚被
   闭合的 A→B→C 路径，是任何单张任务清单都揭示不出来的端到端候选——它正是 `full-chain-testing`
   「通路挖掘」的起点。把它点名列出；路由给 `full-chain-testing`（由它再做关键性分级、只取一小撮
   P0 做安全网）；标注 ⚠️（范围需人工决策），除非已有 CI 端到端闸存在。

5. **据类别路由到对应执行器 skill。** 到 `tool-mapping.md` 查该类的能力缺口 + 路由去向，
   并如实带上状态（✅执行器已建 / 🔧占位待建）。**不在本 skill 写死具体单栈工具**——单后端
   路由给 `backend-testing`，由它按栈（读 `package.json` / `pyproject.toml` / `go.mod` 等）
   解析具体工具；其余类别路由给对应类别 skill（当前占位待建）。

## 优雅降级

- 没有范围标签 → 从任务文本推断类别；注明这是推断。
- 没有依赖图 → 你仍可判定单后端 / 单前端 / 局部前后端；把完整功能链路（及跨模块契约候选类）
  标注"信息不足（依赖图不可用）"。
- 没有契约声明 → 不启用跨模块契约候选类，或将其标注"信息不足"；建议团队把契约文档化，以便
  未来的路由能工作。
- 没有验收标准 → 完整功能链路仍可标记一条链路在 *结构上* 已补全，但要注明具体的旅程步骤尚未
  规约化。

绝不为某个缺失的信号抛错。一份带着诚实"信息不足"标记的部分报告，才是正确的输出。
