# 06-架构基线决策

> 决策日期：2026/05/25
> 决策方式：法庭式对抗调研（15 个候选项目 × 7 维度深挖 + 45 条红队质疑 + 15 条复合挑战 + 代言人逐条回应）
> 决策角色：Lead（法官），基于"幸存论点"综合判决

---

## 一、决策摘要

### 1.1 核心结论

**没有任何一个候选项目值得被直接 fork。**

全部 15 个候选项目经 45 条红队质疑（带源码/issue/commit 硬证据）+ 15 条集成评估师复合挑战的洗礼后，**所有代言人均从"单 fork 最优"立场让步**，承认各自的 owns 项目存在致命缺陷（License 风险、维护死亡、架构缺陷、技术栈冲突等）。

**最终架构基线：自建骨架 + 双项目萃取（PanWatch + Vibe-Trading）**

| 层级 | 策略 | 来源 |
|------|------|------|
| **骨架** | 自建 FastAPI + React 19 + ClickHouse/Redis/PostgreSQL | 全自建（参考各项目设计） |
| **前端基座** | 萃取 PanWatch 的 React 18 + shadcn/ui Dashboard 组件体系 | 借鉴（不 fork） |
| **AI/回测内核** | 萃取 Vibe-Trading 的 Agent 编排 + 回测引擎 + Alpha 因子库 | 借鉴（不 fork） |
| **数据/推送层** | 萃取 daily_stock_analysis 的多源融合 + 全渠道推送封装 | 借鉴（不 fork） |

### 1.2 为什么不是"单 fork"

红队质疑在辩论中未被反驳、被代言人**承认**的关键事实：

| 项目 | 被承认的致命缺陷 | 辩论来源 |
|------|-----------------|---------|
| aiagents-stock | **License 为 null**（GitHub API 404），不是 MIT；94 个 .py 文件全堆根目录 | analyst-ai 承认：降级为"功能逻辑参考" |
| TradingAgents | A股股价错误 12+ open issues 未修，官方默认"想用 A股请自己 fork"；LangGraph 依赖是定时炸弹 | analyst-ai 承认：降级为"架构参考书" |
| go-stock | **GPL-3.0** 传染性 License；Go+Wails 与 Python/Web 方向零兼容 | analyst-ai 承认：工程价值≈0 |
| daily_stock_analysis | fork/star=96.6% 是 GitHub Actions 强制 fork，非社区认可；批处理架构非实时 | analyst-ai 撤回"stars=代码质量"论 |
| Vibe-Trading | A股 DataProvider 不足（官方承认）；不执行实盘 | analyst-ai 接受"技术栈匹配≠功能完备" |
| RQAlpha | **非商业 License**，商业使用需授权 | analyst-quant 未反驳 |
| shares | **16 个月零提交**，已死项目 | analyst-frontend 未反驳 |

| ZVT | SQLAlchemy ORM 与 ClickHouse 列存天然冲突 | analyst-quant 承认 |

| Pan1Watch | 继承 PanWatch 所有架构债；MCP 生态不成熟 | analyst-frontend 承认 |

### 3.2 全自建为什么不选

红队主张"全自建 61 人日比 fork 路线少 16 人日"，但该论点在辩论中被以下事实削弱：

1. **工作量重估的水分**：红队将"阅读 RQAlpha/qteasy 文档总结 A股规则"估算为 0.5 天，实际理解 T+1/涨跌停/停牌/复权等规则的工程实现需要 2-3 人日
2. **前端开发低估**：红队估前端 15 人日，但实际 TradingView Lightweight Charts + shadcn/ui + 预警 Dashboard + PWA 适配需要 18-25 人日
3. **"零技术债"是伪命题**：自建项目同样有技术债（架构迭代、需求变更、bug 修复），只是形式不同

**最终判决**：全自建（75 人日）不是最优，因为可以**安全地借鉴** PanWatch 和 Vibe-Trading 的设计（两者均为 MIT License，读设计不学代码无版权风险），将工作量压缩至 **45-55 人日**（约 2-2.5 个月 1 人全职），同时保留 100% 代码可控性。

## 四、Open Questions（给 brainstorming 阶段）

1. **PanWatch 单进程阻塞架构 bug 的自研替代方案**：如果选择借鉴 PanWatch 设计但自研代码，事件循环应该用 asyncio + uvloop 还是 celery + redis？哪种更适合 7×24 盯盘场景的长连接保持？

2. **Vibe-Trading 452 个 Alpha 因子的取舍**：实际盯盘助手需要多少 Alpha 因子？是全部引入（增加复杂度）还是精选 10-20 个（可能遗漏有效信号）？

3. **A股实时行情的数据源容灾策略**：AkShare 东方财富反爬升级后稳定性下降，是否需要自建本地 tick 缓存（Redis/ClickHouse）+ 多源 failover（Tushare/BaoStock）？缓存策略如何设计？

4. **MCP Server 的长期价值**：Pan1Watch 的 MCP 封装虽不成熟，但 2026 年 Claude Desktop/Cursor 等工具正在快速普及。是否应在 V1 架构中预留 MCP 接口（JSON-RPC 封装），而非 V2 再补？

5. **实盘交易的路径**：Vibe-Trading 明确不执行实盘，aiagents-stock 的 MiniQMT 接口是唯一候选实盘方案但 License 为 null。实盘功能是否推迟到 Phase 3（6 个月后），先聚焦"盯盘+预警+简报"？

### 5.2 与全自建和 fork 路线的对比

| 方案 | 工作量 | 风险 | 可控性 |
|------|--------|------|--------|
| **全自建** | 75 人日 | 低 | 100% |
| **本方案（自建骨架+萃取）** | **45-55 人日** | 中低 | 95%+ |
| 单 fork PanWatch | 35-45 人日 + 技术债 | 高（架构 bug） | 低 |
| 单 fork Vibe-Trading | 40-50 人日 + 缺前端/预警 | 中（A股不足） | 中 |
| fork daily_stock_analysis | 35-55 人日 + 缺实时 | 中（批处理错位） | 中 |

> 注：本方案工作量 45-55 人日是"融合成本"（基于双项目已有设计），比全自建 75 人日节省 20-30 人日；总工作量 84-134 人日是"从零到一"的完整开发（含所有模块），与 05-决策汇总的 75 人日口径一致（仅核心模块）。
