## 一、为什么 / 做什么 / 不做什么

### 1. 业务目标 + 产品目标 + Non-Goals

**业务目标**
- 为个人投资者提供一个可私有部署的 A 股盯盘助手，替代手工刷新行情和手动设预警
- 通过 AI 简报降低信息筛选成本，让用户在开盘前 5 分钟掌握关键市场动态

**产品目标**
- V1（MVP，0-1.5 个月）：自选股管理 + 分钟级行情监控 + 价格/涨跌幅预警 + 飞书推送
- V2（增强，1.5-3 个月）：AI 早盘简报 + 异动检测 + 多源数据融合（可升级 Tushare Pro）
- V3（进阶，3-6 个月）：Agent 编排 + 回测引擎 + PWA 移动端

**Non-Goals（明确不做）**
- ❌ 不替代券商交易软件（无下单功能，06 中所有候选项目均不执行实盘或 License 风险）
- ❌ 不做社交/社区（与雪球差异化，聚焦"帮我盯"而非"给我看"）
- ❌ 不做 Level-2 行情（成本过高，Wind/iFinD 级别非个人场景）
- ❌ 不做高频量化交易（零成本方案 AkShare 延迟 1-5 分钟，Tick 级无法保障）
- ❌ 不做面向他人的数据服务（规避 Tushare 商用边界模糊风险）

### 2. 主画像 + 反画像

**主画像 1：上班族散户**
- 30-40 岁，有本职工作，无法实时盯盘
- 持仓 5-20 只股票，希望"被动通知"而非主动刷新
- 使用同花顺/东方财富但觉得信息过载
- 技术接受度中等，日常用飞书/微信

**主画像 2：轻量量化爱好者**
- 25-35 岁，有编程基础或兴趣
- 想尝试量化策略验证但不愿投入 Wind/QMT 资金门槛
- 需要低成本、可私有部署的实验环境
- 关注可扩展性，未来可能自研策略

**反画像（明确不服务）**
- ❌ 专业日内交易者（需要 Tick 级实时行情 + 专业交易终端）
- ❌ 机构投资者（需要合规授权、多用户权限、Wind 级别数据）
- ❌ 完全不懂技术的用户（无法完成 Docker 部署配置）

| # | 用户故事 | 价值 |
|---|---------|------|

### 5. 每个核心功能的关键边界条件

| 功能 | 数据缺失怎么办 | 并发/规模限制 | 失败怎么办 |
|------|--------------|-------------|----------|
| **自选股管理** | 导入 CSV 格式错误时，返回具体失败行号 + 错误原因 | 单用户场景，MVP 暂不支持多用户 | 导入事务失败回滚，保留上次成功状态 |
| **实时行情** | AkShare 限流/封 IP 时，自动降级到 BaoStock 缓存数据；均不可用时显示"数据暂不可用" | 单进程 asyncio，支持 50-100 只同时轮询（受 AkShare ~3次/秒限流约束） | 连续 3 次获取失败，推送"数据源异常"通知至飞书 |
| **预警检测** | 无最新行情时暂停检测，数据源恢复后补检（避免误报） | 50-100 只同时监控，检测周期 1-5 分钟 | 推送通道失败 → 本地日志记录 → Telegram Bot 兜底 |
| **飞书推送** | — | 免费额度 10,000 次/月；按 20 只×每日触发 3 次×30 天 = 1,800 次，额度充足 | 连续 2 次推送失败，自动切换 Telegram Bot 备用通道 |
| **AI 简报** | LLM API 调用失败时，**重试 2 次**，仍失败则降级为 Jinja2 纯模板简报（无 AI 解读） | 每日 1 次生成，单次调用 ~$0.005 | 生成失败时推送"简报生成失败，请查看 Dashboard" |
| **自然语言设预警** | NLP 解析失败时，提示"未能理解，请用'股票名+条件'格式重试" | — | 解析置信度 < 0.7 时拒绝创建，避免误规则 |

## 三、怎么算做好了

### 6. 非功能需求（性能 / 可用性 / 合规）

**性能（参考 06 架构基线的真实能力）**

| 指标 | 目标值 | 依据 |
|------|--------|------|
| 页面首屏加载 | < 3 秒 | 2核4G 轻量服务器 + React 19 + Vite 构建 |
| 行情数据刷新周期 | 1-5 分钟 | AkShare 零成本方案的实际延迟（非可控） |
| 预警检测延迟 | < 30 秒（从数据获取到推送） | asyncio + Redis Pub/Sub 支撑 |
| 同时监控股票数 | 50-100 只 | 单进程 asyncio + AkShare ~3次/秒限流约束 |
| 简报生成耗时 | < 60 秒 | GPT-4o-mini 单次调用 + Jinja2 模板渲染 |
| 数据库查询（自选股列表）| < 200ms | PostgreSQL + Redis 缓存热点数据 |

**可用性**

| 指标 | 目标值 | 依据 |
|------|--------|------|
| 服务可用性 | ≥ 95% | 个人项目，非商业 SLA；依赖 AkShare 稳定性 |
| 数据源故障自动切换 | < 5 分钟 | APScheduler 定时检测 + facade 自动降级 |
| 7×24 运行 | ✅ | Docker Compose + 服务器长期运行 |
| 推送通道可用性 | ≥ 99% | 飞书 + Telegram 双通道冗余 |

**合规**

| 要求 | 实现 |
|------|------|
| 不直接爬取东方财富/雪球 | 仅通过 AkShare/BaoStock 等封装库间接使用 |
| 不对外提供数据服务 | 系统仅服务部署者本人，规避 Tushare 商用边界 |
| 免责声明 | 简报/预警仅为信息展示，不构成投资建议 |

### 7. 每个用户故事的验收标准（Given/When/Then）

## 四、还不确定的

### 9. 北极星指标 + 关停线

**北极星指标**

| 指标类型 | 指标名称 | 目标值 | 计算方式 |
|----------|---------|--------|----------|
| ⭐ 主指标 | **日均有效预警触达率** | > 90% | 实际触发并成功推送的预警数 / 应触发的预警数 × 100% |
| 辅指标 1 | 日均活跃自选股数 | > 10 只/用户 | 每日至少被刷新过 1 次的自选股数量 |
| 辅指标 2 | 早盘简报打开率 | > 70% | 打开简报查看详情的次数 / 收到简报推送次数 × 100% |
| 辅指标 3 | 数据源可用性 | > 80% | AkShare 成功返回数据的时间 / 交易时段总时间 × 100% |

**关停线（参考 06 架构基线的扩展上限）**

| 条件 | 阈值 | 触发动作 |
|------|------|----------|
| 数据源可用性持续不达标 | < 70% 持续 2 周 | 评估是否升级至 Tushare Pro；若升级后仍不达标，项目暂停 |
| 运营成本超出个人承受范围 | 单用户月均 > 100 元（含服务器 + 数据 + LLM）| 降级至纯 AkShare 零成本方案或暂停 |
| 预警系统失效 | 连续 30 天无任何预警触发 | 排查数据源/检测逻辑故障；修复无果则关停 |
| 架构扩展瓶颈 | 自选股 > 200 只时预警延迟 > 2 分钟 | 说明单进程 asyncio 架构已达上限，需重构为分布式（Celery + Redis）|

| 问题 | 决策 |
