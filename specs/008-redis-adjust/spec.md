# Feature Specification: Dashboard 性能优化

**Feature Directory**: `specs/008-redis-adjust/`  
**Created**: 2026-06-10  
**Status**: Draft  
**Input**: PRD v1.0 Section 5.1 F5（基础 Dashboard）+ `specs/008-redis-adjust/brainstorming.md`  
**Related Feature**: `specs/006-dashboard/`（本优化为 Dashboard 的性能增强，非独立功能）

---

## 1. 功能概述

### 1.1 What

本功能对 MVP 阶段已完成的 **Dashboard 首页（F5）** 进行性能优化，将首响应时长从当前的 2-4 秒（有缓存时不稳定）降低到 **有缓存时 < 1 秒、冷启动 < 2 秒**，同时降低服务端资源消耗和前端刷新时的无效流量。

优化聚焦六大维度：
1. 行情数据缓存化（消除每次请求都实时调用外部接口的瓶颈）
2. 数据库查询并行化（消除串行查询的累积延迟）
3. 刷新端点数据裁剪（消除 Partial 请求中冗余数据的查询和渲染）
4. 历史数据落盘异步化（消除同步写入阻塞主响应）
5. 前端刷新 ETag 支持（消除数据无变化时的重复渲染和传输）
6. 首屏骨架屏（消除用户感知的空白等待）

### 1.2 Why

Dashboard 是用户每天打开最频繁的页面（PRD §3.1：交易日每天 2-3 次），是系统的"门面"。当前 MVP 实现存在以下用户体验问题：

- **每次打开都卡滞**：无论是否重复访问，页面加载时间都在 2-4 秒之间波动，用户无法建立"快速查看"的心智模型
- **刷新浪费资源**：60 秒定时刷新每次都返回完整 HTML，即使数据无变化也消耗服务端渲染和传输资源
- **数据源波动直接影响体验**：外部行情接口（AkShare/BaoStock）响应时间不可控（200ms-2s），直接叠加到首屏，用户感知为"系统慢"
- **无加载反馈**：首屏空白等待期间用户无法判断是网络问题还是系统故障

**业务价值**：将 Dashboard 从"可用但慢"提升到"流畅可用"，验证"低密度信息+主动推送"模式的用户接受度（PRD §2.2 学习目标）。性能是留存的关键门槛——如果用户每次打开都等待 3 秒以上，会逐渐放弃主动查看，完全依赖推送，削弱 Dashboard 的存在价值。

### 1.3 非目标（明确不做）

- ❌ **不改 Dashboard 的页面结构或视觉设计** —— 所有 UI 模块、布局、交互保持与 006-dashboard 一致
- ❌ **不改其他 feature（F1-F4）的业务逻辑** —— 仅优化与 Dashboard 直接相关的数据读取路径
- ❌ **不做全局架构重构** —— 不替换 SQLite 主数据库、不改造 APScheduler 定时任务架构
- ❌ **不做移动端响应式优化** —— MVP 桌面端假设不变（constitution §XII）
- ❌ **不引入独立 Worker 进程（如 Celery）** —— 异步化使用现有线程池/协程机制
- ❌ **不做缓存预热或预聚合** —— 缓存由首次请求自然填充，不增加后台定时填充逻辑

## 2. 用户场景与验收标准

### User Story 1 — 重复打开 Dashboard 快速查看行情（Priority: P0）

作为上班族散户，当我重复打开 Dashboard 时，我希望在 1 秒内看到大盘指数和自选股行情，而不是每次都要等待 3 秒以上。

**Why this priority**: 这是性能优化的核心场景，覆盖 90% 以上的用户访问。有缓存时的快速响应是用户对系统"流畅度"的直观判断。

**Acceptance Scenarios**:

1. **Given** 用户在 5 分钟内已打开过 Dashboard 且系统正常运行，**When** 再次打开 Dashboard，**Then** 页面从请求发送到完整渲染完成的时间 < 1 秒（Chrome DevTools Network 总耗时）
2. **Given** 用户有 50 只自选股，**When** 打开 Dashboard，**Then** 首响应时间仍 < 1 秒，自选股列表完整展示所有 50 只的价格和涨跌幅
3. **Given** Dashboard 已加载完成，**When** 等待 60 秒后自动刷新触发，**Then** 如果行情数据无变化，刷新耗时 < 50ms（服务端返回 304，不重新渲染）；如果有变化，刷新耗时 < 300ms

### User Story 2 — 首次打开或缓存失效时不至于过慢（Priority: P0）

作为用户，当我首次使用系统或后台服务重启后首次打开 Dashboard 时，我希望等待时间控制在可接受范围内，而不是卡死或超时。

**Why this priority**: 冷启动是用户信任系统的关键时刻。如果首次打开就超时或等待过久，用户可能直接放弃。

**Acceptance Scenarios**:

1. **Given** Redis 缓存为空（服务重启后或首次部署），**When** 打开 Dashboard，**Then** 页面在 2 秒内完成加载，展示大盘指数和自选股行情（即使数据来自实时接口）
2. **Given** 外部行情接口响应较慢（如 AkShare 超时），**When** 打开 Dashboard，**Then** 系统返回上次可用的缓存数据（如有）并显示"数据更新延迟"提示，不阻塞页面渲染
3. **Given** 外部行情接口完全不可用且 Redis 缓存为空，**When** 打开 Dashboard，**Then** 页面在 1 秒内返回，自选股区域展示"--"占位符，大盘指数区域展示空状态

### User Story 3 — 数据刷新过程不闪屏不丢失状态（Priority: P1）

作为用户，当 Dashboard 自动刷新行情数据时，我希望页面平滑更新，不闪屏、不丢失滚动位置、不重置分组切换状态。

**Why this priority**: 性能优化不能以牺牲用户体验为代价。如果刷新导致页面闪烁或状态丢失，用户感知比慢更差。

**Acceptance Scenarios**:

1. **Given** Dashboard 已加载且用户滚动到自选股列表中部，**When** 60 秒自动刷新触发且数据有变化，**Then** 只有价格和涨跌幅数字更新，页面不整体重载，滚动位置保持不变
2. **Given** 用户已切换自选股分组到"持仓"，**When** 自动刷新触发，**Then** 刷新后分组仍显示"持仓"，不重置为默认分组
3. **Given** 数据源从不可用恢复为可用，**When** 下一次刷新触发，**Then** 数据自动更新为最新值，延迟提示自动消失，刷新过程不闪屏

### User Story 4 — 服务端资源不被无效刷新耗尽（Priority: P1）

作为系统运维者（部署者本人），当 Dashboard 在前台持续打开时，我希望无效刷新（数据无变化时的刷新请求）不消耗服务端渲染资源，避免长期运行后系统变慢。

**Why this priority**: MVP 单进程架构资源有限（PRD §7.1：内存 < 1GB），减少无效计算能提升系统整体稳定性和并发能力。

**Acceptance Scenarios**:

1. **Given** Dashboard 持续打开 1 小时（触发 60 次刷新），且期间行情数据无变化，**When** 查看服务端日志，**Then** 所有刷新请求均返回 304（无变化），服务端不进行模板渲染和数据库查询
2. **Given** 同时打开 3 个 Dashboard 标签页，**When** 各标签页独立刷新，**Then** 服务端能正确处理 ETag 比对，不因并发请求导致资源耗尽或响应错误

## 3. 功能需求

### FR-001: 行情数据必须支持缓存优先读取模式

系统 MUST 在读取自选股行情和大盘指数时，优先从内存缓存获取；缓存未命中时，才调用外部实时接口。缓存命中时外部接口调用 MUST NOT 发生。

**关键约束 — 对现有调用方透明**：`quote_service` 的缓存行为 MUST 通过参数控制（如 `use_cache: bool`），现有调用方（F3 预警检测定时任务等）默认不启用缓存优先，保持实时读取；Dashboard 层调用时显式启用缓存。此设计确保缓存优化不影响其他 feature 的数据实时性。

**输入**: `stock_codes`: string[]（自选股代码列表）；`use_cache`: bool（是否优先读缓存，默认 False）  
**输出**: `quotes`: [{code, name, price, change_pct, change_amount, update_time}]，与现有格式一致  
**边界**: 缓存未命中时自动回源，回源失败时返回降级数据（空列表或上次可用缓存）；`use_cache=False` 时行为与优化前完全一致

### FR-002: Dashboard 数据聚合 MUST 并行化

系统 MUST 在聚合 Dashboard 首页数据时，对相互独立的数据查询（预警记录、推送历史、通道状态）使用并行方式执行，而非串行顺序执行。

**输入**: 无（内部聚合逻辑）  
**输出**: DashboardViewResponse（与现有格式一致）  
**边界**: 单个查询失败不影响其他查询的返回；并行查询数量不超过 5 个

### FR-003: 行情刷新端点 MUST 裁剪非行情数据

系统 MUST 提供专用的行情刷新端点（`/market_data`），该端点只返回大盘指数和自选股行情相关数据，不查询也不返回预警记录、推送历史、通道状态、简报等不随每次刷新变化的数据。

**输入**: 无（GET 请求）  
**输出**: Partial HTML（大盘指数 + 自选股快照）  
**边界**: 裁剪后的端点 MUST 保证行情数据完整性和格式与首页一致

### FR-004: 历史行情数据落盘 MUST 异步化

系统 MUST 在获取实时行情后，将历史数据落盘操作从主请求线程移出，改为后台异步执行。主请求的响应 MUST NOT 等待落盘完成。

**输入**: `quotes`: 清洗后的行情数据  
**输出**: 无（后台操作，仅记录日志）  
**边界**: 异步落盘失败 MUST NOT 影响主请求的返回；失败时记录 error 日志，不抛异常到用户层

### FR-005: 行情刷新 MUST 支持 ETag/304 机制

系统 MUST 在行情刷新端点支持 HTTP ETag 机制：基于行情数据核心字段生成唯一标识（ETag），前端在后续请求中携带 If-None-Match 头；服务端比对无变化时返回 304（无 body）。

**ETag 计算范围**: 仅基于行情数据的核心字段（大盘指数点位/涨跌幅、各自选股价格/涨跌幅）的规范化字符串的 MD5 hash。不包含更新时间戳、HTML 模板结构、非行情模块数据。此设计确保只有真实行情变化才触发 200，模板微调或时间戳变化不触发重渲染。

**输入**: `If-None-Match`: string（前端携带的上次 ETag）  
**输出**: 304（无变化）或 200 + HTML + 新 ETag  
**边界**: ETag 计算 MUST 轻量（基于数据 hash），不引入显著 CPU 开销；缓存未命中时正常返回 200；ETag 计算异常时降级为正常 200 响应

### FR-006: Dashboard 首屏 MUST 展示骨架屏

系统 MUST 在 Dashboard 首屏渲染时，先展示骨架屏占位（大盘指数骨架卡片 + 自选股表格骨架行 + 简报卡片骨架），数据到达后局部替换为真实内容。

**输入**: 无（模板层面）  
**输出**: 首屏 HTML（含骨架屏标记）  
**边界**: 骨架屏样式 MUST 与真实内容布局一致，避免替换时产生布局跳动；骨架屏在数据到达后（无论成功或失败）立即替换，不设置固定展示时长；数据加载异常时骨架屏替换为降级提示（非永久空白）；服务端 SSR 场景下骨架屏作为初始 HTML 的一部分返回，数据通过后续渲染填充

### FR-007: 缓存失效和降级策略 MUST 可靠

系统 MUST 在外部数据源不可用或缓存服务异常时，自动降级为现有逻辑（直接查数据库或调外部接口），不因为缓存层故障导致功能不可用。

**输入**: 各种异常状态（Redis 不可用、外部接口超时等）  
**输出**: 降级后的正常响应  
**边界**: 降级过程 MUST 对用户透明（除提示条外，页面正常展示）；降级时性能目标放宽（< 3s 可接受）

## 4. 成功标准

### Measurable Outcomes

| ID | 指标 | 目标值 | 测量方式 |
|:---|:---|:---|:---|
| SC-001 | 有缓存时首响应 | < 1s | Chrome DevTools Network → 从请求发出到 DOMContentLoaded |
| SC-002 | 冷启动首响应（缓存为空） | < 2s | 重启缓存服务后首次访问，同上 |
| SC-003 | 行情刷新（数据无变化） | < 50ms | 304 响应的 TTFB |
| SC-004 | 行情刷新（数据有变化） | < 300ms | 200 响应的 TTFB（含渲染） |

| SC-006 | 闪屏/状态丢失 | 0 次 | 人工测试：刷新 20 次，观察是否闪屏或分组重置 |
| SC-007 | 服务端无效渲染次数 | 0 次/小时 | 数据无变化时的刷新请求，服务端模板渲染次数为 0 |
| SC-008 | 降级透明度 | 100% | 所有异常场景下，页面不报错，展示降级提示 + 可用数据 |

## 5. 边界条件与异常处理

### 5.1 数据边界

| 场景 | 行为 |
|------|------|
| 自选股数量为 0 | 展示引导页（与 006-dashboard 一致），性能目标不适用 |
| 自选股数量 1-50 | 完整展示，首响应目标 < 1s（性能基线） |
| 自选股数量 > 50 | 仍完整展示，首响应目标放宽至 < 1.5s；Dashboard 展示上限以 006-dashboard 为准（50 只），超过时分页或折叠 |
| 单只股票停牌 | 正常展示，价格显示为停牌前收盘价，涨跌幅为 0 |
| 推送历史 > 100 条 | 只展示最近 100 条；当前数据规模下全表排序取 100 条耗时可接受（< 100ms）；如后续推送日志表数据量 > 10 万条，需单独做数据归档（不在本优化范围内） |
| 缓存数据过期（> TTL） | 自动回源，回源失败时展示过期数据 + "数据更新延迟"提示 |
| 外部接口返回异常值（如价格为负） | 数据清洗层过滤，该股票显示"--" |
| 缓存 key 命名 | 采用 `quotes:{market|watchlist}` 格式，单用户场景无需用户 ID 前缀；大盘指数 key 为 `quotes:market`，自选股 key 为 `quotes:watchlist:{hash_of_codes}` |
| 缓存数据格式 | JSON 序列化存储，与现有 API 响应数据结构一致 |

### 5.2 异常场景

| 场景 | 行为 |
|------|------|
| Redis 服务不可用 | 自动降级为直接调外部接口 / 查数据库，记录 warning 日志，页面正常展示 |
| 外部行情接口超时 | 返回 Redis 缓存数据（如有），无缓存时返回空数据 + "数据暂不可用"提示 |
| 外部行情接口限流 | 同超时处理，额外在 Dashboard 顶部显示"数据源限流"黄色提示 |
| 异步落盘失败 | 记录 error 日志，不影响主响应，下次行情获取时重试落盘 |
| ETag 计算异常 | 降级为正常 200 响应，不阻塞功能 |
| 并发刷新请求（多标签页） | 各请求独立处理，ETag 比对正常生效，不因并发导致 304 误判 |
| Redis 不可用 + 外部接口超时（双重降级） | 优先返回 SQLite 缓存数据（现有 cache_service），其次返回空数据 + 降级提示；此场景性能目标放宽至 < 3s |
| Redis 不可用 + 外部接口可用 | 正常调外部接口获取实时数据，不经过 Redis；性能与优化前一致 |

### 5.3 MVP 约束

- **单用户架构**：所有缓存 key 无需用户隔离，直接以数据类型命名（constitution §II）
- **零成本数据源**：缓存策略优先服务于零成本数据源的高延迟特性，不为付费数据源预留特殊逻辑（constitution §III）
- **桌面端唯一**：骨架屏和 ETag 机制仅需在桌面端（1280px+）验证（constitution §XII）
- **暗色模式唯一**：骨架屏颜色使用暗色模式设计 token（constitution §X）
- **Redis 是新增层，不改现有 SQLite 缓存**：现有 `cache_service.py` 的 SQLite 缓存架构保持不动，Redis 作为独立的内存缓存层叠加使用；Dashboard 优先使用 Redis，Redis 不可用时自动降级到 SQLite 缓存或直连外部接口

### 5.4 API 边界（限频、失败、超时、配额）

| 维度 | 定义 | 说明 |
|------|------|------|
| **限频** | 前端刷新间隔固定 60s（由 JS 控制），服务端不对 `/market_data` 做额外限频 | MVP 单用户场景，无恶意刷接口风险；如后续多用户化，需补充限频 |
| **超时** | 外部行情接口调用超时设为 3s；超过 3s 视为失败，触发缓存降级 | 3s 为 PRD §7.1 单只股票查询 p95 上限 |
| **失败重试** | 外部接口失败时重试 1 次（共 2 次调用），仍失败则彻底降级 | 避免单次网络抖动导致降级 |
| **配额** | 无 API 配额限制（AkShare/BaoStock 为开源库，无调用配额）；飞书 Open API 配额由 F4 推送模块管理，本优化不涉及 | |
| **ETag 过期** | ETag 与缓存 TTL 绑定（60s），缓存过期后 ETag 自然失效 | 不单独维护 ETag 状态 |
| **并发请求** | 单用户场景下并发请求数上限为浏览器标签页数（通常 < 10），服务端不做连接数限制 | |

## 6. 假设

- **A-001**: 当前 Docker Compose 环境可引入 Redis 服务（新增一个容器），用户接受部署复杂度轻微增加
- **A-002**: 外部行情接口（AkShare/BaoStock）的响应时间在正常网络环境下 < 1s，极端情况 < 3s；缓存 TTL 固定为 60s（与 006-dashboard FR-003 的刷新间隔一致），MVP 阶段不可配置
- **A-003**: Dashboard 的现有模板结构（Jinja2 + Tailwind CSS CDN）不变，骨架屏通过 CSS animation 实现，无需新增 JS 库
- **A-004**: 用户对行情数据的分钟级延迟可接受（PRD §2.2 假设 1），60s 缓存 TTL 不会导致用户抱怨"数据不实时"
- **A-005**: 现有 SQLite 数据库的推送历史表数据量 < 10 万条，全表排序取 100 条的耗时在可接受范围；如后续数据量增长，需单独做数据归档

## 7. 与现有系统的集成假设

本优化为 **006-dashboard 的增强**，而非独立功能。以下集成点假设成立：

| 集成点 | 假设 | 验证方式 |
|--------|------|----------|
| F1 自选股管理 | `watchlist_service.get_watchlists()` 接口不变，返回格式与 006-dashboard 一致 | 单元测试 mock 验证 |

### Session 2026-06-09

- **Q1**: 缓存基础设施是否可引入 Redis？→ **A**: 可以引入 Redis 作为独立容器
- **Q2**: 优化范围是仅 Dashboard 还是可改上游？→ **A**: Dashboard + 上游相关接口局部优化（`quote_service`、`dashboard_service`）

### Session 2026-06-10（Clarify 自审）

- **Q4**: `quote_service` 增加缓存会影响 F3 预警检测的实时性吗？→ **A**: 通过 `use_cache` 参数控制，F3 调用时默认 False，保持实时读取；Dashboard 调用时显式传 True
- **Q5**: Redis 与现有 SQLite `cache_service` 的关系？→ **A**: Redis 是新增独立层，不改 SQLite 缓存架构；降级时优先用 Redis，不可用则回退到 SQLite 缓存或直接调外部接口
- **Q6**: 自选股上限到底多少？→ **A**: Dashboard 展示上限 50 只（与 006-dashboard SC-008 一致），F1 允许添加 100 只；50 只为性能基线，超过时目标放宽
- **Q7**: TTL 60s 固定还是可配置？→ **A**: MVP 阶段固定 60s，不可配置（与 006-dashboard FR-003 刷新间隔一致）
- **Q8**: ETag 计算包含哪些字段？→ **A**: 仅行情核心字段（价格/涨跌幅）的 MD5 hash，不包含时间戳和模板结构
- **Q9**: 推送历史表数据量大时怎么办？→ **A**: 当前数据规模下全表排序取 100 条可接受；> 10 万条时需单独归档（不在本优化范围）
- **Q10**: 双重降级（Redis 不可用 + 外部接口超时）的性能目标？→ **A**: 此极端场景目标放宽至 < 3s，优先返回 SQLite 缓存或空数据+提示
