## User Scenarios & Testing *(mandatory)*

### User Story 1 - 查看低密度行情首页 (Priority: P1)

作为上班族散户，当我打开 Dashboard 时，我希望在一个页面上快速看到大盘指数、自选股价格和最新简报，无需切换多个页面。

**Why this priority**: Dashboard 是用户每天打开最频繁的页面，是系统的"门面"。信息密度过高会加剧信息焦虑，低密度设计能降低决策成本，让用户 30 秒内掌握全局。

**Independent Test**: 单独测试时，不依赖实时行情刷新，直接 mock 自选股和行情数据，验证页面正确渲染所有模块。

**Acceptance Scenarios**:

1. **Given** 用户已添加 5 只自选股且行情数据正常，**When** 打开 Dashboard，**Then** 页面展示：大盘指数（上证指数/深证成指/创业板指）、自选股列表含当前价和涨跌幅、最近一条简报卡片、快捷入口栏
2. **Given** 用户已添加自选股，**When** 查看自选股区域，**Then** 每只股票的卡片展示：股票名称、代码、当前价格、涨跌幅、涨跌额，以及当日价格走势迷你折线图
3. **Given** 用户有多组自选股分组（如"持仓"/"关注"），**When** 点击分组切换，**Then** 自选股列表切换为对应分组的股票，其他模块保持不变
4. **Given** 页面加载完成，**When** 等待 60 秒，**Then** 行情数据自动刷新，价格和涨跌幅更新为最新值，刷新过程不闪屏；数据源异常时停止自动刷新，恢复后自动重启
5. **Given** 今日有股票触发预警，**When** 查看 Dashboard，**Then** 顶部展示"今日预警汇总"区域（含触警股票列表和触发条件），对应股票卡片上显示预警图标（🔴 alert / 🟡 watch）

### User Story 2 - 首次使用引导 (Priority: P1)

作为新用户，当我首次使用系统且没有添加任何自选股时，我希望看到清晰的引导，而不是空白页面。

**Why this priority**: 首次体验决定留存。空白页面会让用户困惑是否配置错误，引导能缩短从"安装完成"到"获得价值"的时间。

**Independent Test**: 单独测试时，清空自选股数据后打开 Dashboard，验证引导页正确展示。

**Acceptance Scenarios**:

1. **Given** 用户首次使用，自选股列表为空，**When** 打开 Dashboard，**Then** 隐藏大盘指数和行情区域，展示"添加自选股"引导卡片，含"批量导入"和"手动添加"两个入口
2. **Given** 用户处于引导页，**When** 点击"手动添加"，**Then** 展开搜索框，支持输入股票名称/代码搜索并添加
3. **Given** 用户成功添加第一只自选股后，**When** 页面刷新，**Then** 引导页消失，展示完整 Dashboard 内容

### User Story 3 - 数据源异常时的降级展示 (Priority: P2)

作为用户，当数据源暂时不可用时，我希望看到上次缓存的数据和明确的提示，而不是报错页面。

**Why this priority**: 零成本数据源（AkShare/BaoStock）在交易高峰期可能不稳定。优雅降级能维持用户信任，避免因临时数据中断而弃用系统。

**Independent Test**: 单独测试时，模拟数据源不可用，验证页面展示缓存数据和提示信息。

**Acceptance Scenarios**:

1. **Given** 数据源当前不可用，但有 30 分钟内的缓存数据，**When** 打开 Dashboard，**Then** 展示缓存的行情数据，并在顶部显示"数据更新延迟"黄色提示条
2. **Given** 数据源不可用且无缓存数据，**When** 打开 Dashboard，**Then** 各行情区域显示"--"占位符，展示"数据暂不可用，请稍后刷新"提示
3. **Given** 数据源从不可用恢复为可用，**When** 自动刷新触发，**Then** 数据更新为最新值，延迟提示条自动消失

---

### User Story 4 - 移动端自适应访问 (Priority: P2)

作为上班族散户，当我用手机访问 Dashboard 时，我希望页面自适应为适合手机屏幕的单列布局。

**Why this priority**: 用户画像中明确 PC + 手机双设备使用。移动端是碎片化时间（午休、通勤）查看行情的主要场景，自适应是基本体验要求。

**Independent Test**: 单独测试时，使用浏览器模拟手机 viewport 打开 Dashboard，验证布局正确切换。

**Acceptance Scenarios**:

1. **Given** 用户使用手机屏幕访问（viewport < 768px），**When** 打开 Dashboard，**Then** 页面切换为单列布局，大盘指数横向滚动展示，自选股卡片纵向排列，隐藏快捷入口栏（折叠为底部导航）
2. **Given** 用户在移动端查看自选股，**When** 点击某只股票卡片，**Then** 展开显示更多信息（当日最高/最低/成交量）
3. **Given** 用户在 PC 端和移动端分别访问，**When** 对比两端的布局，**Then** 核心信息（大盘+自选股+简报）在两端均完整展示，仅排列方式不同

### User Story 5 - 查看推送历史与通道状态 (Priority: P2)

作为用户，当我需要排查为什么没收到推送时，我希望在 Dashboard 上查看最近的推送记录和通道健康状态。

**Why this priority**: 推送是预警的"最后一公里"，用户没收到消息时需要自助排查。推送历史是系统可信度的体现。

**Independent Test**: 单独测试时，mock 推送日志数据，验证 Dashboard 正确展示推送历史和状态。

**Acceptance Scenarios**:

1. **Given** 过去 24 小时有 10 条推送记录（8 成功 2 失败），**When** 用户查看推送历史区域，**Then** 列表展示：消息类型、股票/简报标题、发送时间、通道、状态、失败原因
2. **Given** 飞书通道当前不可用（API 限流/连续错误），**When** 用户查看 Dashboard，**Then** 在通道状态区域显示"主通道限流/不可用，已切换备用通道"红色提示
3. **Given** 某条推送因配置错误失败，**When** 用户点击失败记录，**Then** 展开详情展示失败原因和修复建议

---

### Edge Cases

- 自选股数量超过 50 只 → 分页或折叠展示，默认只显示前 20 只，提供"展开全部"
- 单只股票涨跌幅超过 ±10%（科创板/创业板）→ 涨跌幅颜色按规则显示（红涨绿跌），不做特殊截断
- 大盘指数数据延迟超过 15 分钟 → 显示"数据延迟"警告，而非静默展示旧数据
- 用户同时打开多个浏览器标签页 → 各标签页独立刷新，不强制同步
- 浏览器离线 → 展示"网络异常"提示，保留上次加载的静态内容
- 简报内容为空（非交易日或首次使用）→ 简报区域显示"今日暂无简报"占位卡片
- 推送日志超过 100 条 → 仅展示最近 100 条，提供"查看更多"入口

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供 Web Dashboard 首页，展示大盘指数、自选股、简报三大核心模块
- **FR-002**: 系统 MUST 支持自选股按分组切换显示，分组来自 F1 自选股管理
- **FR-003**: 系统 MUST 支持行情数据定时自动刷新（固定 60 秒间隔，不可配置），刷新过程不闪屏；数据源异常时停止自动刷新，恢复后自动重启
- **FR-004**: 系统 MUST 在首次使用无自选股时展示引导页，引导用户添加股票
- **FR-005**: 系统 MUST 在数据源不可用时展示缓存数据 + 明确的延迟提示，无缓存时展示占位符
- **FR-006**: 系统 MUST 支持移动端自适应布局（viewport < 768px 切换单列）
- **FR-007**: 系统 MUST 在 Dashboard 上展示最近 7 天的推送历史记录，含状态过滤
- **FR-008**: 系统 MUST 在 Dashboard 上展示推送通道健康状态（飞书/Telegram 可用性）
- **FR-009**: 系统 MUST 支持搜索框入口，可跳转至股票添加/预警设置（搜索框本身在 Dashboard，具体功能由 F1/F3 提供）
- **FR-010**: 系统 MUST 在自选股卡片上展示：股票名称、代码、当前价、涨跌幅、涨跌额

### Key Entities

- **DashboardView（Dashboard 视图）**: 首页展示单元。核心属性包括：视图 ID、用户标识、布局模式（desktop/mobile）、最后刷新时间、数据版本。
- **MarketSnapshot（大盘快照）**: 当前市场指数数据。核心属性包括：指数名称、当前点位、涨跌幅、涨跌额、更新时间。
- **StockCard（股票卡片）**: 自选股展示单元。核心属性包括：股票代码、股票名称、当前价格、涨跌幅、涨跌额、所属分组、当日价格走势迷你折线图数据点序列、更新时间。
- **PushHistoryWidget（推送历史组件）**: 推送记录展示单元。核心属性包括：消息类型、股票/简报标题、发送时间、通道、状态、失败原因。
- **ChannelStatusWidget（通道状态组件）**: 通道健康展示单元。核心属性包括：通道名称、状态（active/degraded/unavailable）、限流状态、最后更新时间。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户从打开页面到看到完整 Dashboard 的加载时间 < 3 秒（首次加载，含网络请求）
- **SC-002**: 用户从打开页面到看到缓存 Dashboard 的加载时间 < 1 秒（非首次，本地渲染）
- **SC-003**: 行情数据刷新间隔 60 秒，刷新过程中页面不闪屏、不丢失滚动位置
- **SC-004**: 移动端自适应后，核心信息（大盘+自选股+简报）完整展示，无需横向滚动（大盘指数区域除外）
- **SC-005**: 数据源异常时，用户 100% 能看到明确的降级提示（而非空白或报错）
- **SC-006**: 推送历史查询响应时间 < 500ms（最近 100 条）
- **SC-007**: 首次使用引导页让新用户在 2 分钟内完成第一只自选股添加
- **SC-008**: Dashboard 同时展示的自选股上限为 50 只，超过时分页或折叠，页面不卡顿

---

## Assumptions

- **A-001**: Dashboard 前端使用 HTMX + Jinja2 模板渲染（由项目架构基线决定），MVP 不做 SPA 框架
- **A-002**: 自选股数据和分组由 F1（自选股管理）提供 API，Dashboard 只负责展示
- **A-003**: 行情数据由 F2（数据源 Facade）提供，Dashboard 从缓存读取（不直接调数据源）
- **A-004**: 预警状态由 F3（实时行情与预警）提供，Dashboard 展示已触发预警的标记
- **A-005**: 推送历史由 F4（推送通知）提供，Dashboard 调用其查询 API
- **A-006**: AI 简报内容在 v1.0 MVP 阶段为固定模板（由 F3 定时任务生成），v1.1 接入 AI
- **A-007**: Dashboard 不处理用户认证，MVP 阶段为单用户系统，无登录流程
- **A-008**: 移动端自适应通过 CSS 媒体查询实现，不做单独的移动端应用或 PWA
- **A-009**: 搜索框在 MVP 阶段为保留占位：显示搜索框 UI，点击后提示"自然语言搜索即将上线"，不实现搜索功能，跳转添加股票/设置预警通过其他入口
- **A-010**: 快捷入口在 MVP 阶段固定为 4 个（选股/预警/复盘/设置），不支持用户自定义
- **A-011**: 复盘功能在 MVP 阶段仅展示当日预警触发记录和历史行情对比，详细复盘留到 v1.1
