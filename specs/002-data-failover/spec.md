## User Scenarios & Testing *(mandatory)*

### User Story 1 - 主数据源故障时自动切换 (Priority: P1)

作为系统运维者，我希望当主数据源（AkShare）因限流或接口变动导致请求失败时，系统能自动切换到备用数据源（BaoStock）继续提供服务，而不需要我手动干预。

**Why this priority**: 这是零成本数据源方案的核心保障。AkShare 作为开源免费库，存在限流、封 IP、接口变动的风险。如果没有自动切换，数据源一旦故障，整个盯盘系统（行情、预警、简报）将全部停摆，产品失去基本可用性。

**Independent Test**: 单独测试时，模拟主数据源故障后，系统仍能成功获取数据并返回正确的数据源标识，无需其他功能配合。

**Acceptance Scenarios**:

1. **Given** 主数据源 AkShare 正常可用，**When** 系统发起行情查询请求，**Then** 返回 AkShare 数据，状态标记为 "primary"
2. **Given** 主数据源 AkShare 返回超时错误，**When** 系统发起行情查询请求，**Then** 自动尝试 BaoStock 备用源，成功后返回数据并标记 "fallback"，同时触发"数据源切换"通知
3. **Given** 主数据源连续 3 次返回错误，**When** 第 4 次发起请求，**Then** 系统不再尝试 AkShare，直接使用 BaoStock，AkShare 被标记为不可用状态
4. **Given** 主数据源已标记为不可用，**When** 定时健康检查检测到 AkShare 连续 2 次成功，**Then** 自动将 AkShare 恢复为可用状态，后续请求优先使用 AkShare

### User Story 2 - 全部数据源故障时提供缓存数据 (Priority: P1)

作为盯盘用户，即使所有数据源都不可用，我也希望系统能展示最近一次的缓存数据，而不是直接报错或空白页面，让我至少知道之前的行情状态。

**Why this priority**: 全部数据源同时故障的概率虽低，但一旦发生（如网络中断、数据源同时维护），用户面对的是完全不可用的系统。提供缓存数据可以维持基础体验，避免用户因"完全看不到数据"而放弃使用。

**Independent Test**: 单独测试时，断开所有外部数据源后，系统仍能返回最近一次成功获取的数据，并明确提示数据为缓存。

**Acceptance Scenarios**:

1. **Given** AkShare 和 BaoStock 均返回错误，**When** 系统发起行情查询请求，**Then** 返回本地缓存数据（最近 1 小时内），状态标记为 "cached"，Dashboard 显示"数据更新延迟"提示
2. **Given** 全部数据源不可用且本地缓存已过期（超过 1 小时），**When** 系统发起请求，**Then** 返回空数据，状态标记为 "unavailable"，Dashboard 显示"数据暂不可用"
3. **Given** 系统处于缓存数据模式，**When** 任意数据源恢复可用，**Then** 后续请求自动使用恢复的数据源，不再返回缓存

---

### User Story 3 - 数据源状态可观测 (Priority: P2)

作为系统运维者，我希望在 Dashboard 上看到当前各数据源的实时健康状态，以及最近一次切换的时间点和原因，以便判断是否需要手动调整数据源配置。

**Why this priority**: 自动切换虽然不需要手动干预，但运维者需要了解系统正在发生什么。如果数据源频繁切换，可能意味着主数据源稳定性恶化，需要考虑升级方案。可观测性是运维的基础需求。

**Independent Test**: 单独测试时，可以通过 Dashboard 或日志查看数据源的健康状态和切换历史，无需触发实际的数据请求。

**Acceptance Scenarios**:

1. **Given** 数据源发生切换，**When** 用户查看 Dashboard 数据源状态区，**Then** 看到当前主备源的健康状态（可用/不可用）、当前使用的数据源标识、最近一次切换时间
2. **Given** 主数据源因连续失败被标记为不可用，**When** 用户查看系统日志，**Then** 看到完整的失败记录：每次失败的时间、错误类型（超时/限流/返回错误）、切换决策
3. **Given** 数据源在交易时段（9:30-11:30, 13:00-15:00）发生故障，**When** 系统触发切换，**Then** 推送"数据源异常"通知到用户主通道（飞书），提示当前数据来源

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 支持配置至少 2 个数据源：主数据源（AkShare）和备用数据源（BaoStock）
- **FR-002**: 系统 MUST 在每次数据获取请求时优先使用主数据源
- **FR-003**: 系统 MUST 在主数据源请求失败（超时、限流、返回错误码）时，自动降级到备用数据源
- **FR-004**: 系统 MUST 在主数据源连续失败达到 3 次时，将其标记为不可用，后续请求不再尝试该源，直到健康检查恢复
- **FR-005**: 系统 MUST 在主备数据源均失败时，返回本地缓存数据（最近 1 小时内有效），标记状态为 "cached"
- **FR-006**: 系统 MUST 在主备数据源均失败且缓存过期时，返回空数据，标记状态为 "unavailable"
- **FR-007**: 系统 MUST 每 5 分钟执行一次定时健康检查，探测各数据源可用性
- **FR-008**: 系统 MUST 在健康检查中，对标记为不可用的主数据源连续 2 次成功后才恢复其可用状态，避免抖动
- **FR-009**: 系统 MUST 在数据源发生切换时（主→备、备→主、任意→缓存），记录切换日志，包含时间、原因、目标数据源
- **FR-010**: 系统 MUST 在交易时段内发生数据源降级（主→备 或 任意→缓存）时，推送"数据源异常"通知到用户
- **FR-011**: 系统 MUST 向调用方返回实际使用的数据源标识（"primary" / "fallback" / "cached" / "unavailable"）
- **FR-012**: 系统 MUST 在 facade 层预留数据类型参数接口，支持未来扩展不同数据类型的容灾策略。MVP 阶段统一按实时行情策略处理所有请求

### Key Entities

- **DataSource（数据源）**: 代表一个外部数据提供方。核心属性包括：名称（如 "akshare" / "baostock"）、类型（实时行情/历史数据/基本面）、健康状态（可用/不可用）、连续失败计数、最近一次成功时间、最近一次失败时间、失败原因。数据源由系统管理员在配置文件中定义，运行时可切换。
- **DataFetchRequest（数据获取请求）**: 代表一次对外部数据的调用。核心属性包括：请求的数据类型、目标股票代码列表、请求发起时间、超时时间。请求经过数据源 facade 层处理，调用方不直接感知底层数据源切换。
- **DataFetchResult（数据获取结果）**: 代表一次数据获取的完整响应。核心属性包括：实际数据内容、使用的数据源标识、状态（"primary" / "fallback" / "cached" / "unavailable"）、响应时间、是否为缓存数据。所有调用方统一接收此结果对象。
- **CacheEntry（缓存条目）**: 代表一条已缓存的数据。核心属性包括：缓存键（数据类型+股票代码）、缓存内容、缓存时间、过期时间（默认 1 小时）。缓存持久化到 SQLite 的 `cache_entries` 表，进程重启后不丢失。
- **HealthCheckRecord（健康检查记录）**: 代表一次定时健康检查的结果，仅作为结构化日志格式定义，不持久化到数据库。核心属性包括：检查时间、数据源名称、检查结果（成功/失败）、响应时间、错误信息。通过应用日志输出，保留 30 天。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 主数据源故障时，系统自动切换到备用源的延迟 < 5 分钟（从首次失败到完成切换）
- **SC-002**: 数据源切换过程中，调用方（如行情模块、预警模块）无感知，请求成功率 >= 99%
- **SC-003**: 全部数据源不可用时，缓存数据的有效期 >= 1 小时，缓存命中率 >= 90%
- **SC-004**: 主数据源恢复后，自动切回的检测时间 < 15 分钟（连续 2 次成功 + 5 分钟检查间隔）
- **SC-005**: 数据源健康状态的可观测性：Dashboard 状态刷新延迟 < 30 秒
- **SC-006**: 数据源切换通知的触达率：交易时段内降级事件通知成功率 >= 95%
- **SC-007**: 数据获取请求的总响应时间（含切换）：p95 < 15 秒（含备用源重试 + 缓存读取）

## Clarifications

### Session 2026-05-26

- **Q1**: MVP 阶段数据类型容灾策略的范围？
  - **A**: 仅实时行情容灾（Option A）。MVP 阶段 facade 层统一按实时行情策略处理所有数据类型请求，历史数据和基本面数据请求失败时直接报错，不切换、不缓存。FR-012 修改为"预留数据类型参数接口，v1.1 扩展多类型容灾"。
- **Q2**: 缓存数据是否应持久化到 SQLite？
  - **A**: SQLite 持久化缓存（Option B）。缓存写入 SQLite 的 `cache_entries` 表，含 key/content/timestamp/expires_at 字段。进程重启后缓存不丢失，与自选股管理共享数据库。A-003 已更新。
- **Q3**: 健康检查记录是否需要持久化保存？
  - **A**: 仅日志输出，不持久化（Option A）。健康检查结果写入应用日志（结构化 JSON），不保存到数据库。Dashboard 仅展示当前实时状态。HealthCheckRecord 实体从持久化模型中移除，仅作为日志结构定义。日志保留 30 天，与 PRD 审计日志要求对齐。

## Assumptions

- **A-001**: MVP 阶段仅支持 AkShare（主）和 BaoStock（备）两个数据源，不支持用户自定义更多数据源
- **A-002**: 数据源的健康检查通过发送一个轻量级探测请求（如查询单只股票快照）来实现，不使用独立的心跳接口
- **A-003**: 缓存数据持久化到 SQLite（与自选股管理共享数据库），通过 `cache_entries` 表存储，含 key/content/timestamp/expires_at 字段。MVP 阶段不使用 Redis 等外部缓存服务
- **A-004**: 缓存的有效期固定为 1 小时，不支持按数据类型配置不同有效期
- **A-005**: 数据源切换的通知仅推送一次（同一次降级事件不重复通知），恢复主源时不推送恢复通知
- **A-006**: 数据源配置（URL、超时时间、重试次数）通过环境变量或配置文件管理，不支持运行时热修改
- **A-007**: 健康检查的 5 分钟间隔是固定的，不支持按交易时段/非交易时段区分频率
- **A-008**: 数据源返回的"限流"错误通过 HTTP 429 状态码或异常类型识别，不同数据源的限流信号格式可能不同，需要在 facade 层统一映射
