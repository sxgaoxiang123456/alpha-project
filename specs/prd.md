## 2. 项目背景与目标 (Project Background & Goals)

### 2.1 背景

A 股个人投资者面临的信息过载问题日益严重。同花顺 MAU 超 3500 万，东方财富股吧日均发帖 800 万+，但海量信息反而增加了决策成本。上班族散户群体（30-40 岁，有本职工作）无法实时盯盘，依赖券商 APP 推送又常因信息过载而关闭通知。

现有产品的核心矛盾：
- **同花顺/东方财富**：功能全面但信息密度过高，用户需要主动筛选
- **雪球**：社区驱动，但社交内容分散注意力
- **券商 AI 产品**（AI涨乐/广发易淘金）：体验先进但绑定单一券商生态
- **开源工具**：多处于批处理/回测/教学状态，缺少实时盯盘+AI简报的组合

市场上缺乏一款「可私有部署、低成本、聚焦盯盘+预警+简报」的独立工具。

### 2.2 目标

**业务目标**
- 为个人投资者提供一个可私有部署的 A 股盯盘助手，替代手工刷新行情和手动设预警
- 通过 AI 简报降低信息筛选成本，让用户在开盘前 5 分钟掌握关键市场动态

**产品目标**
- V1（MVP，0-1.5 个月）：自选股管理 + 分钟级行情监控 + 价格/涨跌幅预警 + 推送
- V2（增强，1.5-3 个月）：AI 早盘简报 + 异动检测 + 多源数据融合
- V3（进阶，3-6 个月）：Agent 编排 + 回测引擎 + PWA 移动端

**学习目标**
- 验证「低密度信息+主动推送」模式能否降低散户的信息焦虑
- 验证零成本数据源（AkShare + BaoStock）能否支撑个人级盯盘需求

### 2.3 非目标 (Non-Goals)

- ❌ **不替代券商交易软件**：不提供下单/交易功能，仅做信息展示与预警
- ❌ **不做社交/社区**：与雪球差异化，聚焦"帮我盯"而非"给我看"
- ❌ **不做 Level-2 行情**：成本过高，非个人场景可承受
- ❌ **不做高频量化交易**：零成本方案延迟 1-5 分钟，Tick 级无法保障
- ❌ **不做面向他人的数据服务**：规避数据源商用边界模糊风险
- ❌ **不做投资建议**：所有输出仅为信息展示，不构成投资建议

## 3. 目标用户与画像 (Target Users & Personas)

### 3.1 主要画像 P1：上班族散户

| 维度 | 描述 |
|------|------|
| **背景** | 30-40 岁，有本职工作，非职业投资者 |
| **目标** | 在不盯盘的情况下，不错过持仓股票的关键异动 |
| **痛点** | ① 工作时间无法看行情；② 同花顺/东方财富信息过载；③ 手动设预警步骤繁琐；④ 错过买卖时机后悔 |
| **使用频次** | 交易日每天 2-3 次（早盘、午休、收盘后） |
| **设备** | PC（上班时）+ 手机（碎片化时间） |
| **付费意愿** | 年运营成本 < 1000 元可接受 |
| **技术能力** | 能完成 Docker 基础配置，但不会写代码 |

### 3.2 次要画像 P2：轻量量化爱好者

| 维度 | 描述 |
|------|------|
| **背景** | 25-35 岁，有编程基础或兴趣 |
| **目标** | 低成本验证量化策略想法，需要可扩展的实验环境 |
| **痛点** | ① Wind/QMT 资金门槛高（10-50 万）；② 开源工具多为批处理/回测，缺少实时能力；③ 部署复杂 |
| **使用频次** | 每天多次，周末也会回测 |
| **设备** | PC 为主 |
| **付费意愿** | 年运营成本 < 2000 元可接受 |
| **技术能力** | 会写 Python，熟悉 Linux/Docker |

### 3.3 反画像（明确不服务）

- ❌ **专业日内交易者**：需要 Tick 级实时行情 + 专业交易终端，本系统 1-5 分钟延迟无法满足
- ❌ **机构投资者**：需要合规授权、多用户权限、Wind 级别数据，个人工具无法覆盖
- ❌ **完全不懂技术的用户**：无法完成 Docker 部署和配置文件修改

## 4. 用户故事 / 使用场景 (User Stories & Scenarios)

### US-01 批量导入持仓并分组管理

> 作为上班族散户，
> 我希望批量导入持仓股票并分组管理，
> 以便快速查看我关心的股票动态。

**触发情境**：用户首次使用，或季度调仓后更新持仓列表
**期望结果**：20 只股票在 1 分钟内导入完成，按"持仓"/"观察"/"自研"分组显示

### US-02 价格预警实时推送

> 作为无法盯盘的用户，
> 我想设置价格/涨跌幅预警并在触发时收到推送通知，
> 以便及时响应持仓异动。

**触发情境**：工作开会期间，持仓股票突然大跌
**期望结果**：5 分钟内收到推送（标题+当前价+跌幅），可一键查看详情

### US-03 AI 早盘简报

> 作为信息过载的投资者，
> 我想在每天开盘前收到 AI 生成的早盘简报，
> 以便 5 分钟掌握市场动态。

**触发情境**：每天上班路上/早餐时间，快速了解当日市场
**期望结果**：9:00 前收到结构化简报（大盘概览 + 异动 TOP 5 + 板块热点）

### US-04 自然语言设预警

> 作为不想学习复杂界面的用户，
> 我想用自然语言设置预警（如"茅台跌破 1500 提醒我"），
> 以便快速创建监控规则。

**触发情境**：临时想到某个条件，不想打开表单逐项填写
**期望结果**：输入一句话，3 秒内确认规则创建成功

### US-05 成交量异动发现机会

> 作为轻量量化爱好者，
> 我想系统自动检测成交量突增/价格异动并推送，
> 以便发现潜在机会。

**触发情境**：盘后复盘，寻找次日关注标的
**期望结果**：收盘后收到当日异动汇总，含涨速/成交量/关联板块

## 6. 详细功能描述 (Detailed Feature Specifications)

### F1 自选股管理

**触发**：用户通过 Dashboard 搜索框、CSV 导入、或手动输入添加股票

**输入**：
- `stock_code`: string, A 股 6 位代码（如 "600519"），必填
- `stock_name`: string, 股票名称（如 "贵州茅台"），选填，系统可自动补全
- `group_name`: string, 分组名称（如 "持仓"/"观察"），默认 "默认分组"
- `cost_price`: float, 持仓成本价，选填，用于盈亏展示
- `shares`: int, 持仓股数，选填

**核心流程**：
1. 用户输入代码/名称，系统查询股票基础信息（名称、所属板块）
2. 校验代码有效性（是否存在于 A 股全市场代码表）
3. 校验是否已存在于自选股列表（去重）
4. 保存至用户自选股表，关联分组
5. 触发实时行情订阅（将该股票加入监控列表）

**输出**：
- `success`: bool
- `stock`: { code, name, group, added_at }
- `message`: string, 操作结果提示

**边界条件**：
- 代码不存在 → 返回 "股票代码不存在，请检查输入"
- 代码已存在 → 返回 "该股票已在自选股中，请勿重复添加"
- 自选股达上限（MVP 期 100 只）→ 返回 "自选股已达上限，请先移除其他股票"
- CSV 导入格式错误 → 返回具体失败行号 + 错误原因，已成功的部分保留

**异常处理**：
- 全量导入事务失败 → 回滚至上一次成功状态，不保留部分导入结果
- 数据源查询失败 → 允许用户手动输入名称，标记为"待验证"

### F2 基础实时行情

**触发**：APScheduler 定时任务（1-5 分钟周期）+ 用户打开 Dashboard 时主动刷新

**输入**：
- `stock_codes`: string[], 待查询的股票代码列表
- `fields`: string[], 查询字段（price, change_pct, volume, turnover 等），默认全部

**核心流程**：
1. 调度器触发，获取当前订阅的股票代码列表
2. 调用主数据源获取最新行情快照
3. 数据清洗：格式标准化、异常值检测、缺失值标记
4. 更新实时缓存（最新行情供 Dashboard/预警模块读取）
5. 历史数据异步落盘（供简报生成和回测使用）

**输出**：
- `quotes`: [{ code, name, price, change_pct, change_amount, volume, turnover, update_time }]
- `data_source`: string, 实际使用的数据源标识
- `status`: string, "ok"/"degraded"/"unavailable"

**边界条件**：
- 主数据源限流/封 IP → 自动降级到备用源，标记状态为 "degraded"
- 全部数据源不可用 → 返回上次缓存数据，状态 "unavailable"，推送"数据源异常"通知
- 非交易时段查询 → 返回最近收盘数据，标记 "market_closed"
- 单只股票停牌 → 该股票返回停牌标记，其余正常

**性能要求**：
- 50-100 只股票批量查询 p95 < 10 秒
- 单只股票查询 p95 < 3 秒

### F3 价格/涨跌幅预警

**触发**：定时扫描任务比较最新行情与用户预设规则

**输入**（规则配置）：
- `stock_code`: string, 目标股票
- `condition_type`: enum["price_above", "price_below", "change_pct_above", "change_pct_below", "volume_above"]
- `threshold`: float, 阈值
- `cooldown_minutes`: int, 冷却期（同一规则触发后 N 分钟内不重复推送），默认 30
- `level`: enum["watch", "alert"], 触达级别，默认 "alert"

**核心流程**：
1. 每次行情更新后，遍历所有生效的预警规则
2. 逐条评估：最新行情是否满足条件
3. 检查冷却期：该规则是否在冷却期内？是则跳过
4. 满足条件且不在冷却期 → 触发推送
5. 记录触发日志，启动冷却期计时

**输出**：
- `triggered`: bool
- `alert`: { rule_id, stock_code, condition, triggered_value, triggered_at, level }

**边界条件**：
- 无最新行情时（数据源中断）→ 暂停检测，数据源恢复后补检（避免基于过期数据误报）
- 连续触发（股价恰好在阈值附近波动）→ 冷却期机制防止刷屏
- 规则创建时股票已满足条件 → 不立即触发，等待条件从"不满足"变为"满足"
- 用户同时设置多条规则且同时触发 → 合并为一条推送（单股票单分钟内最多 1 条）

**业务规则**：
- 单用户最多 50 条生效规则（防止过度监控）
- Level 1 "watch"（关注级）：标准推送，无声音
- Level 2 "alert"（预警级）：强提醒推送 + 声音/震动标记
- 冷却期内同一规则不重复触发

### F4 推送通知

**触发**：预警触发 / 简报生成完成 / 系统异常通知

**输入**：
- `channel`: enum["feishu", "telegram"], 推送通道
- `message_type`: enum["alert", "briefing", "system"]
- `payload`: object, 消息内容体
- `priority`: enum["normal", "high"]

**核心流程**：
1. 推送模块接收待发送消息
2. 选择主通道（飞书）尝试推送
3. 主通道失败 → 自动降级到备用通道（Telegram）
4. 双通道均失败 → 本地日志记录，Dashboard 显示未读标记
5. 记录推送日志（成功/失败/耗时）

**输出**：
- `sent`: bool
- `channel_used`: string, 实际使用的通道
- `error`: string, 失败原因（如有）

**边界条件**：
- 飞书 Open API QPS 限流或连续 API 错误 → 自动切换 Telegram，Dashboard 提示"主通道限流/不可用"
- Telegram 需代理但代理不可用 → 本地日志记录，标记为"待重试"
- 推送内容过长 → 截断为摘要+"查看详情"链接
- 网络超时 → 重试 2 次，仍失败则降级

**消息形态**：

**预警卡片示例**：
```
🔴 预警：贵州茅台（600519）跌破 1500 元
─────────────────────────
当前价：1498.50  (-0.10%)
触发条件：价格 < 1500
触发时间：14:32:15
─────────────────────────
[查看详情] [暂停此预警] [调整阈值]
```

**简报卡片示例**：
```
📊 A股早盘简报 | 2026年05月26日
─────────────────────────
【大盘概览】
上证指数 3367.12  +0.23%
深证成指 10172.58  +0.45%
创业板指 2038.91  +0.67%
─────────────────────────
【异动 TOP 3】
🔴 XX股份(600XXX) 涨速+5.2%  半导体
🟡 YY科技(002XXX) 放量突破  新能源
🟢 ZZ医药(603XXX) 主力资金流入  医药
```

### F5 基础 Dashboard

**触发**：用户打开 Web 页面

**输入**：无（展示用户已配置的数据）

**核心流程**：
1. 加载用户自选股列表和分组
2. 从缓存读取最新行情数据
3. 渲染低密度首页：大盘指数 → 自选股速览 → AI 简报卡片 → 快捷入口
4. 建立实时更新连接（轮询或长连接），定时刷新行情

**页面结构**：
```
┌─────────────────────────────────────────┐
│  🔍 搜索框（支持自然语言搜股/设预警）    │
├─────────────────────────────────────────┤
│     上证指数  深证成指  创业板指         │  ← 大字、高对比度
├─────────────────────────────────────────┤
│  【自选股】价格走势速览（可分组切换）    │  ← 右滑切换文字/图表
├─────────────────────────────────────────┤
│  【AI简报】今日异动 TOP 5（预判式卡片）  │  ← 点击展开详情
├─────────────────────────────────────────┤
│  【快捷入口】选股 / 预警 / 复盘 / 设置   │  ← 可自定义拖动
└─────────────────────────────────────────┘
```

**边界条件**：
- 首次使用无自选股 → 展示"添加自选股"引导，隐藏行情区
- 数据源不可用 → 展示上次缓存数据 + "数据更新延迟"提示
- 移动端访问 → 自适应为单列布局，隐藏非核心模块

### F6 数据多源容灾

**触发**：每次数据获取请求 / 定时健康检查

**输入**：
- `preferred_source`: string, 用户偏好的主数据源
- `required_data_type`: enum["realtime", "historical", "fundamental"]

**核心流程**：
1. 尝试主数据源获取数据
2. 主数据源失败（超时/限流/返回错误）→ 标记为不可用，切换备用源
3. 备用源成功 → 返回数据，标记实际使用的源
4. 全部源失败 → 返回缓存数据（如有），标记状态
5. 定时健康检查恢复主数据源 → 自动切回

**输出**：
- `data`: object, 请求的数据
- `source_used`: string, 实际使用的数据源
- `status`: enum["primary", "fallback", "cached", "unavailable"]

**边界条件**：
- 主源连续 3 次失败 → 标记为不可用，触发"数据源切换"通知
- 备用源也失败 → 使用本地缓存（最近 1 小时内的数据）
- 全部不可用 → 返回空数据 + "数据暂不可用"状态
- 切回主源前需验证主源连续 2 次成功，避免抖动

---

### F7 AI 早盘简报（v1.1）

**触发**：APScheduler 定时任务（交易日 8:50）

**输入**：
- `market_data`: 前一日收盘数据 + 当日早盘数据（如有）
- `alert_history`: 过去 24 小时预警触发记录
- `template`: string, 简报结构模板

**核心流程**：
1. 提取关键数据：大盘指数、涨跌停家数、成交额、异动股票 TOP 10
2. 构造 Prompt，调用 LLM API 生成自然语言解读
3. LLM 返回结果 + Jinja2 模板渲染结构化卡片
4. 推送至用户主通道

**输出**：
- `briefing`: string, 结构化简报文本
- `sections`: [{ title, content }]
- `key_stocks`: [{ code, name, change_pct, reason }]

**边界条件**：
- LLM API 调用失败 → 重试 2 次，仍失败则降级为纯 Jinja2 模板简报（无 AI 解读）
- 非交易日 → 跳过生成，记录日志
- 生成耗时超过 60 秒 → 超时中断，降级为模板简报

---

### F8 自然语言设预警（v1.1）

**触发**：用户在搜索框输入自然语言指令

**输入**：
- `query`: string, 如 "茅台跌破 1500 提醒我"

**核心流程**：
1. 解析查询：提取股票名称/代码 + 条件类型 + 阈值
2. 名称模糊匹配到标准代码（如 "茅台" → "600519"）
3. 构建规则对象，校验有效性
4. 创建规则并返回确认

**输出**：
- `success`: bool
- `rule`: { stock_code, stock_name, condition, threshold }
- `message`: string, "预警已创建：贵州茅台 价格 < 1500"

**边界条件**：
- 解析失败（无法理解）→ 提示"未能理解，请用'股票名+条件'格式重试"
- 股票名称歧义（如 "银行" 对应多只股票）→ 列出候选列表让用户选择
- 解析置信度 < 0.7 → 拒绝创建，提示手动配置入口
- 条件格式不支持 → 提示"暂不支持该条件类型，请用价格/涨跌幅条件"

---

### F9 成交量异动检测（v1.1）

**触发**：定时扫描任务（与行情刷新同步）

**输入**：
- `stock_codes`: string[], 监控列表
- `volume_threshold`: float, 成交量突增倍数阈值，默认 2.0
- `change_speed_threshold`: float, 涨速阈值（%），默认 3.0

**核心流程**：
1. 获取最新行情和近期历史数据（5-20 日均价量）
2. 逐只计算：当前成交量 vs N 日均量倍数、N 分钟涨速
3. 满足任一条件 → 标记为异动，记录异动类型
4. 批量生成异动通知（单股票单分钟内最多 1 条）

**输出**：
- `anomalies`: [{ code, name, type, value, threshold, triggered_at }]
- `type`: enum["volume_surge", "price_surge", "both"]

**边界条件**：
- 新股/次新股历史数据不足 → 跳过均量计算，仅用绝对阈值
- 停牌股票 → 跳过检测
- 涨跌停状态 → 涨速检测自动调整（涨停后不再报"涨速过快"）
- 同时满足多条件 → 合并为一条"复合异动"通知

## 9. 验收标准 (Acceptance Criteria)

### AC-F1-01：添加单只自选股

**Given** 用户访问 Dashboard
**When** 在搜索框输入 "600519" 并确认
**Then** 该股票出现在自选股列表，显示名称"贵州茅台"和最新价格，且 3 秒内完成

### AC-F1-02：批量导入自选股

**Given** 用户持有包含 20 只股票的 CSV 文件
**When** 上传文件并选择"持仓"分组
**Then** 20 只股票全部导入成功，Dashboard 显示该分组，导入耗时 < 10 秒

### AC-F1-03：自选股去重

**Given** 用户已添加 "600519"
**When** 再次尝试添加 "600519"
**Then** 系统提示"该股票已在自选股中"，列表不重复

**Given** 用户有 20 只自选股
**When** 打开 Dashboard
**Then** 30 秒内展示所有股票最新价格和涨跌幅，数据时间戳为最近 5 分钟内

### AC-F2-02：数据源降级

**Given** 主数据源不可用
**When** Dashboard 加载行情
**Then** 自动展示备用源数据，页面显示"数据来自备用源"提示，不展示错误页

### AC-F3-01：价格预警触发

**Given** 用户为 "600519" 设置 "价格 < 1500" 预警
**When** 最新行情价格 = 1498
**Then** 5 分钟内用户收到推送卡片（含股票名、当前价、触发条件）

### AC-F3-02：冷却期防刷屏

**Given** "600519" 预警已触发且处于 30 分钟冷却期
**When** 价格再次 < 1500
**Then** 不推送第二次通知，Dashboard 记录日志"冷却期内跳过"

**Given** 飞书认证失败（app_id/app_secret 配置错误）或 API 限流
**When** 预警触发
**Then** 自动尝试 Telegram 通道，Telegram 成功则用户收到通知；双失败则 Dashboard 显示未读标记

### AC-F4-01：简报生成与推送

**Given** 交易日 8:50
**When** 系统执行简报生成任务
**Then** 9:00 前用户收到飞书卡片，包含上证指数/深证成指/创业板指 + 至少 3 条异动信息

### AC-F4-02：简报降级

**Given** LLM API 连续 3 次调用失败
**When** 简报生成任务执行
**Then** 降级为纯模板简报（无 AI 解读），用户收到结构化数据卡片而非错误通知

### AC-F5-01：Dashboard 低密度展示

**Given** 用户有 15 只自选股
**When** 打开 Dashboard
**Then** 首屏展示：大盘指数（3 个）+ 自选股前 6 只 + 1 张简报卡片，无信息过载感

### AC-NFR-01：性能验收

**Given** 100 只自选股 + 50 条预警规则
**When** 行情刷新周期
**Then** 全量检测耗时 < 30 秒，系统资源占用（内存）< 1GB

## 11. 度量指标 (Metrics)

### 11.1 北极星指标

**日均有效预警触达率** = 实际触发并成功推送的预警数 / 应触发的预警数 x 100%
- 目标值：> 90%

### 11.2 关键指标

**Acquisition（获取）**
- 暂无（个人工具，无公开注册）

**Activation（激活）**
- 首次添加自选股 <= 3 只的用户占比 < 20%（防止体验后放弃）
- 完成推送配置的用户占比 > 80%

**Retention（留存）**
- 7 日活跃（至少触发 1 次行情刷新）> 60%
- 30 日活跃 > 40%

**Engagement（参与）**
- 日均活跃自选股数 > 10 只/用户
- 早盘简报打开率 > 70%
- 预警推送点击率 > 30%

**Data Quality（数据）**
- 数据源可用性（交易时段成功返回率）> 80%
- 推送通道成功率 > 99%

### 11.3 反指标（防止"指标作弊"）

- 自选股数量平均 >= 5（防止只为完成新手任务添加后不再使用）
- 预警触发后用户 24 小时内打开 Dashboard 查看 > 20%（防止推送被忽视）
- 简报生成中纯模板降级占比 < 30%（防止 LLM 频繁失效）

### 11.4 度量频率

- **日报**：预警触达率 / 数据源可用性 / 推送成功率
- **周报**：7 日留存 / 活跃自选股数 / 简报打开率
- **月报**：完整指标回顾 + 数据源稳定性评估
