# Feature Specification: 飞书 lark-cli 环境配置

**Feature Directory**: `specs/007-lark-cli-config/`  
**Created**: 2026-06-08  
**Status**: Draft  
**Input**: MVP v1.0 遗留推送通知问题：将飞书主通道从 webhook 配置调整为 lark-cli / 飞书 Open API 配置；凭证只读 `.env`，复用现有推送链路，设置页不再收集 webhook。

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 使用 .env 配置飞书主通道 (Priority: P1)

作为私有部署用户，当我已经在环境变量文件中提供飞书应用凭证和群聊标识时，我希望系统自动使用这些配置作为飞书主推送通道，让预警消息能发送到指定飞书群。

**Why this priority**: 飞书是推送通知的主通道，也是预警价值交付的最后一公里。当前 webhook 配置与 lark-cli 发送链路不一致，导致主通道无法完成真实发送；优先修复配置来源才能恢复 MVP 闭环。

**Independent Test**: 单独测试时，准备完整的飞书环境变量并触发一条测试推送，验证系统选择飞书主通道并记录发送结果。

**Acceptance Scenarios**:

1. **Given** 飞书应用凭证和群聊标识已在环境变量文件中配置完整，**When** 系统发送预警推送，**Then** 推送优先走飞书主通道，并将消息发送到对应飞书群
2. **Given** 飞书配置完整且主通道发送成功，**When** 用户查看推送历史，**Then** 该条记录显示为飞书通道发送成功，并包含发送耗时
3. **Given** 飞书主通道发送失败但备用通道可用，**When** 系统重试后仍失败，**Then** 自动降级到备用通道，并保留主通道失败原因

---

### User Story 2 - 设置页不再误导用户填写 webhook (Priority: P1)

作为用户，当我打开系统设置页时，我希望看到飞书主通道由环境变量配置的明确说明，而不是继续看到已经无效的 webhook 输入框。

**Why this priority**: webhook 输入框会误导用户以为填写机器人 webhook 即可恢复推送，但实际发送链路需要飞书应用凭证。移除错误配置入口能减少排障成本，避免用户保存无效配置。

**Independent Test**: 单独测试时，打开设置页，验证页面不再出现飞书 webhook 表单输入项，且清楚说明飞书主通道的配置来源。

**Acceptance Scenarios**:

1. **Given** 用户打开系统设置页，**When** 查看推送通道区域，**Then** 页面展示飞书主通道由环境变量配置的说明
2. **Given** 用户保存系统设置，**When** 表单提交完成，**Then** 系统不再新增或更新飞书 webhook 配置
3. **Given** 飞书配置未完整提供，**When** 用户查看设置页，**Then** 页面能提示飞书主通道未完整配置，且不展示任何敏感凭证值
4. **Given** 用户修改环境变量文件中的飞书配置，**When** 用户查看设置页提示，**Then** 页面说明配置变更需要重启服务后生效

---

### User Story 3 - 保持现有推送降级与日志体验 (Priority: P2)

作为用户，当飞书主通道配置缺失或发送失败时，我希望现有 Telegram 备用通道、本地日志和 Dashboard 排障信息继续可用，不因为主通道配置调整影响其他推送能力。

**Why this priority**: 本次调整只聚焦飞书主通道配置接线，不能破坏已经完成的推送日志、备用通道和 Dashboard 状态展示。保持现有体验能降低回归风险。

**Independent Test**: 单独测试时，分别模拟飞书配置缺失、飞书发送失败、备用通道可用与不可用，验证推送日志和降级结果符合既有行为。

**Acceptance Scenarios**:

1. **Given** 飞书配置缺失但 Telegram 已配置，**When** 系统发送推送，**Then** 系统不尝试使用无效飞书配置，并按既有规则记录或降级
2. **Given** 飞书和 Telegram 均不可用，**When** 系统发送推送，**Then** 本地推送日志保留失败记录，便于用户在 Dashboard 排查
3. **Given** Telegram 配置已存在，**When** 本 feature 完成后保存设置页其他配置，**Then** Telegram 配置和发送行为不受影响
4. **Given** 飞书发送所需运行时能力缺失，**When** 系统尝试发送飞书推送，**Then** 失败原因能被记录为可排查的客户端不可用类问题，不阻塞系统其他功能

---

### Edge Cases

- 飞书环境变量只配置了一部分 → 视为主通道未完整配置，不展示密钥明文，不使用半配置发送
- 飞书环境变量完整但凭证错误 → 主通道发送失败并记录认证失败原因，按既有规则降级或记录失败
- 飞书群聊标识错误或应用未入群 → 主通道发送失败并记录失败原因，用户可通过推送历史排查
- 飞书命令行工具不可用 → 主通道发送失败并记录客户端错误，不影响系统启动和本地日志记录
- 用户历史数据库中仍存在旧 webhook 配置 → 不再作为飞书主通道配置来源，也不在设置页继续展示为可编辑项
- 用户只修改 Telegram 或系统偏好设置 → 不应清空或覆盖飞书环境变量配置状态
- 用户修改 `.env` 后未重启服务 → 设置页应提示环境配置读取具有启动时边界，避免误判为保存失败
- 用户界面需要显示配置状态 → 只能显示完整/缺失和必要的非敏感字段状态，不显示应用密钥明文

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 将飞书主通道的配置来源限定为环境变量文件中的飞书应用凭证和群聊标识
- **FR-002**: 系统 MUST 在飞书应用凭证和群聊标识完整时，将飞书主通道视为可尝试发送的主通道
- **FR-003**: 系统 MUST 在飞书配置缺失或不完整时，不使用旧 webhook 配置作为替代主通道
- **FR-004**: 系统 MUST 复用现有推送主备通道规则：飞书优先、失败重试、备用通道降级、本地日志兜底
- **FR-005**: 系统 MUST 保留现有 Telegram 配置方式和发送行为，不把 Telegram 纳入本次配置变更范围
- **FR-006**: 系统 MUST 在设置页移除飞书 webhook 可编辑输入入口，避免用户保存无效 webhook 配置
- **FR-007**: 系统 MUST 在设置页展示飞书主通道由环境变量配置的只读说明，并能表达配置是否完整
- **FR-008**: 系统 MUST 避免在页面、日志和错误提示中展示飞书应用密钥等敏感值
- **FR-009**: 系统 MUST 在推送历史中保留飞书主通道发送成功、失败、降级所需的状态和失败原因
- **FR-010**: 系统 MUST 保持现有行情、预警检测、Dashboard 展示、数据源配置和系统偏好行为不变
- **FR-011**: 系统 MUST 在飞书发送所需运行时能力缺失时记录可排查的失败原因，并继续保留备用通道和本地日志兜底
- **FR-012**: 系统 MUST 向用户说明飞书环境变量配置需要通过部署环境维护，并在服务重新加载配置后生效
- **FR-013**: 系统 MUST 对历史遗留 webhook 配置采取只忽略不迁移的策略，不自动删除用户已有数据库记录

### Key Entities

- **FeishuRuntimeConfig（飞书运行配置）**: 表示运行时可用的飞书主通道配置状态。核心属性包括：应用标识是否存在、应用密钥是否存在、群聊标识是否存在、配置完整性。
- **FeishuDiagnosticState（飞书诊断状态）**: 表示用户可排查的飞书通道状态。核心属性包括：配置完整性、最近一次失败分类、是否疑似运行时能力缺失、修复提示。
- **PushChannelStatus（推送通道状态）**: 表示飞书和备用通道的健康状态。核心属性包括：通道名称、可用状态、连续失败情况、最近更新时间。
- **PushLog（推送日志）**: 表示每次推送的结果记录。核心属性包括：消息类型、使用通道、发送状态、失败原因、耗时。
- **SettingsPushChannelView（设置页推送通道视图）**: 表示设置页展示给用户的推送通道配置状态。核心属性包括：飞书配置来源说明、配置完整性、Telegram 可编辑配置项。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在飞书环境配置完整且凭证有效时，预警推送能够优先通过飞书主通道送达，成功记录率达到 100%
- **SC-002**: 设置页中飞书 webhook 可编辑入口数量降为 0，用户不会再通过页面保存飞书 webhook
- **SC-003**: 飞书应用密钥在用户界面和普通错误提示中的明文展示次数为 0
- **SC-004**: 飞书配置缺失或发送失败时，推送日志记录成功率达到 100%，不丢失待推送消息结果
- **SC-005**: 备用通道已配置且可用时，飞书主通道失败后的降级发送成功率保持既有目标 100%
- **SC-006**: 本次调整后，现有 Telegram 配置、数据源偏好、刷新间隔、告警冷却设置均可继续保存和读取
- **SC-007**: 用户在 1 分钟内可从设置页理解飞书主通道应通过环境变量配置，而非 webhook 输入框
- **SC-008**: 飞书发送所需运行时能力缺失时，相关失败记录 100% 包含可用于排障的失败分类
- **SC-009**: 历史 webhook 配置在本 feature 完成后不会被用于任何新的飞书主通道发送决策

---

## Assumptions

- **A-001**: 飞书 `app_id`、`app_secret`、`brand`、`chat_id` 已由部署者提供在环境变量文件中，本 feature 不负责在页面采集这些值
- **A-002**: 本 feature 只修复飞书主通道配置来源和运行时接线，不改变推送消息格式、重试次数、降级策略和日志模型
- **A-003**: Telegram 继续作为可选备用通道，由现有设置页配置，不纳入本次主通道配置调整
- **A-004**: 历史遗留的 webhook 配置即使存在，也不再参与飞书主通道发送决策
- **A-005**: 运行环境应具备调用飞书发送工具的能力；若运行环境缺失该能力，应作为发送失败被记录，而不是阻塞系统启动
- **A-006**: 本 feature 不修改既有 feature 文档，不扩展行情、预警、Dashboard 或部署体系的功能范围
- **A-007**: 环境变量配置属于部署期配置；用户修改环境变量文件后，需要重启或重新加载服务才能保证新值被运行时使用
- **A-008**: 本 feature 不自动删除旧 webhook 数据库记录，避免无授权清理历史配置；旧记录仅不再被飞书主通道读取
