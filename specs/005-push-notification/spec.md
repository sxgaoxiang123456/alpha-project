## User Scenarios & Testing *(mandatory)*

### User Story 1 - 收到预警推送卡片 (Priority: P1)

作为盯盘用户，当我的预警规则被触发时，我希望在 5 分钟内收到飞书推送卡片，卡片上展示股票名称、当前价格、触发条件和触发时间，并能一键查看详情。

**Why this priority**: 推送是预警的"最后一公里"——预警检测再准确，如果用户收不到或看不懂推送，价值为零。飞书卡片形态（结构化+可交互）比纯文本推送体验好得多。

**Independent Test**: 单独测试时，不依赖行情刷新和预警检测，直接调用推送模块发送一条预警卡片，验证用户收到正确的飞书消息。

**Acceptance Scenarios**:

1. **Given** 预警 "600519 价格 < 1500" 已触发，**When** 系统发送推送，**Then** 用户 5 分钟内收到飞书卡片，含：股票名称、当前价 1498.50、触发条件、触发时间、"查看详情"按钮
2. **Given** 预警 Level 为 "alert"（预警级），**When** 推送发送，**Then** 卡片含红色标记/紧急标识，与 "watch" 级区分
3. **Given** 飞书 Open API 配置正确且未触发限流，**When** 推送发送，**Then** 状态标记为 "sent"，记录实际耗时
4. **Given** 预警卡片内容超长（如合并了 3 条规则的摘要），**When** 推送发送，**Then** 自动截断为摘要+"查看详情"链接，不截断关键信息（股票名/价格/触发条件）

### User Story 2 - 推送通道降级与备用通道 (Priority: P1)

作为用户，当飞书 API 调用失败时，我希望系统自动尝试 Telegram 备用通道，确保我不会因为主通道故障而错过重要预警。

**Why this priority**: 飞书 Open API 受 QPS 限流约束（企业自建应用约 20 次/秒），且 API 可能因配置错误/网络问题/限流失败。没有备用通道，主通道一旦故障用户就完全收不到通知，产品可靠性大打折扣。

**Independent Test**: 单独测试时，模拟飞书推送失败，验证系统自动降级到 Telegram；模拟双通道都失败，验证本地日志记录。

**Acceptance Scenarios**:

1. **Given** 飞书 API 返回 500 错误或触发限流，**When** 推送发送，**Then** 自动重试 2 次仍失败后，切换到 Telegram 通道发送
2. **Given** 飞书 API 限流或连续错误，**When** 推送发送，**Then** 自动降级到 Telegram，Dashboard 提示"主通道限流/不可用"
3. **Given** 飞书和 Telegram 双通道均不可用，**When** 推送发送，**Then** 本地记录推送失败日志，Dashboard 显示未读标记，不丢失推送内容
4. **Given** Telegram 未配置或代理不可用，**When** 推送尝试，**Then** 本地记录失败日志，不阻塞主流程；MVP 阶段 Telegram 为可选通道

### User Story 3 - 收到早盘简报推送 (Priority: P1)

作为用户，我希望在交易日开盘前收到 AI 生成的早盘简报推送，包含大盘指数、异动 TOP 3 和板块热点。

**Why this priority**: 简报是"被动盯盘"的补充——用户不需要自己看行情，系统主动推送市场概览。这是产品"自动化信息交付"定位的核心体现。

**Independent Test**: 单独测试时，直接调用推送模块发送一条简报卡片，验证用户收到正确格式。

**Acceptance Scenarios**:

1. **Given** 简报生成完成（大盘指数 + 异动股票），**When** 系统发送简报推送，**Then** 用户收到飞书卡片，含：日期、上证指数/深证成指/创业板指、异动 TOP 3、"查看完整简报"按钮
2. **Given** 简报内容超长，**When** 推送发送，**Then** 截断为摘要，保留大盘指数和 TOP 3，隐藏详细分析
3. **Given** 飞书不可用但 Telegram 可用，**When** 发送简报，**Then** 降级到 Telegram 文本格式（简报卡片降级为纯文本摘要）

---

### User Story 4 - 查看推送历史与状态 (Priority: P2)

作为用户，我希望在 Dashboard 上查看最近的推送历史，了解哪些预警已发送、哪些失败了、为什么失败。

**Why this priority**: 推送历史是排查问题的关键——用户没收到推送时，需要知道是规则没触发、推送失败了、还是通道配置有问题。推送历史也是产品可信度的体现。

**Independent Test**: 单独测试时，查询推送历史 API，验证返回最近 N 条推送记录及状态。

**Acceptance Scenarios**:

1. **Given** 系统过去 24 小时发送了 10 条推送（8 成功 2 失败），**When** 用户查看推送历史，**Then** 列表展示：消息类型、股票/简报标题、发送时间、通道、状态、失败原因
2. **Given** 某条推送因飞书认证失败，**When** 用户查看详情，**Then** 展示失败原因"飞书 API 返回 403 或认证错误"，提示"检查飞书 app_id/app_secret 配置"
3. **Given** 推送历史超过 30 天保留期限，**When** 用户查询，**Then** 仅展示 30 天内的记录，过期记录每日凌晨自动清理

---

### Edge Cases

- 飞书卡片"查看详情"按钮 → 跳转 Dashboard 对应页面；"暂停预警"/"调整阈值"按钮留到 v1.1
- 推送内容超长（超过飞书卡片 4096 字符限制）→ 智能截断，保留关键信息
- 网络超时 → 重试 2 次（指数退避），仍失败则降级
- 双通道均失败 → 本地持久化记录，Dashboard 显示未读标记
- 短时间内大量推送（如 10 条规则同时触发）→ 批量合并为一条消息，避免刷屏
- 非交易日推送简报 → 跳过，记录日志
- 飞书 API 连续错误率超过阈值（> 90% 请求失败）→ Dashboard 提示"主通道限流/即将不可用"
- Telegram Bot Token 过期 → 标记为不可用，后续直接跳过 Telegram

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 支持通过 lark-cli 调用飞书 Open API 发送推送卡片（主通道）
- **FR-002**: 系统 MUST 支持通过 Telegram Bot API 发送推送消息（备用通道）
- **FR-003**: 系统 MUST 实现推送通道自动降级：主通道失败 → 自动重试 2 次 → 仍失败则切换到备用通道
- **FR-004**: 系统 MUST 在双通道均不可用时，本地记录推送失败日志，Dashboard 显示未读标记
- **FR-005**: 系统 MUST 支持三种消息类型：alert（预警）、briefing（简报）、system（系统异常）
- **FR-006**: 系统 MUST 对预警消息生成结构化卡片，包含：股票名称/代码、当前价格、涨跌幅、触发条件、触发时间、操作按钮
- **FR-007**: 系统 MUST 对简报消息生成结构化卡片，包含：日期、大盘指数、异动 TOP 3、板块热点摘要
- **FR-008**: 系统 MUST 对超长推送内容自动截断，保留关键信息（股票名/价格/触发条件/日期/大盘指数），超出部分用"查看详情"替代
- **FR-009**: 系统 MUST 记录每次推送日志：消息类型、目标通道、发送时间、状态（sent/failed/pending）、失败原因、耗时
- **FR-010**: 系统 MUST 提供推送历史查询 API，支持按时间范围、消息类型、状态过滤
- **FR-011**: 系统 MUST 支持推送内容按优先级处理：high 优先发送，normal 排队
- **FR-012**: 系统 MUST 在飞书 API 连续限流/错误时自动标记主通道为不可用，后续推送直接走备用通道

### Key Entities

- **PushMessage（推送消息）**: 待发送的消息实体。核心属性包括：消息 ID、消息类型（alert/briefing/system）、优先级（normal/high）、内容体（结构化数据）、目标通道、创建时间。
- **PushLog（推送日志）**: 已发送/失败的推送记录。核心属性包括：日志 ID、消息 ID、通道（feishu/telegram）、状态（sent/failed/pending）、失败原因、发送耗时、创建时间。
- **PushChannel（推送通道状态）**: 通道的健康状态和配置。核心属性包括：通道名、状态（active/degraded/unavailable）、配置参数（app_id/app_secret/brand/chat_id / Bot Token）、连续失败次数、限流状态。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 推送从提交到用户收到的延迟 p95 < 5 分钟（预警级）/ < 10 分钟（简报级）
- **SC-002**: 主通道故障时自动降级到备用通道的成功率 = 100%（备用通道可用时）
- **SC-003**: 双通道均失败时的本地记录成功率 = 100%，不丢失任何推送内容
- **SC-004**: 推送历史查询响应时间 p95 < 500ms（最近 100 条）
- **SC-005**: 飞书卡片到达率 >= 99%（配置正确且未触发限流时）
- **SC-006**: 用户可在 Dashboard 查看最近 7 天的推送历史及状态
- **SC-007**: 推送内容截断后关键信息完整率 = 100%（股票名、价格、触发条件不丢失）
- **SC-008**: 预警级（alert）推送在 5 分钟内到达率 >= 95%

---

## Assumptions

- **A-001**: 飞书 app_id、app_secret、brand、chat_id 和 Telegram Bot Token 由用户在设置页配置，本模块不管理配置 UI
- **A-002**: 预警/简报的内容生成由上游模块（F3/F7）负责，本模块仅负责格式化渲染和发送
- **A-003**: 飞书 Open API 受 QPS 限流约束（企业自建应用约 20 次/秒），超出限流或连续 API 错误时降级到 Telegram
- **A-004**: Telegram 需要代理访问，代理配置由用户设置
- **A-005**: 简报推送在 v1.0 MVP 阶段由 quote_scheduler 在交易日 9:00 触发固定模板简报（大盘指数 + 自选股概览），v1.1 接入 AI 后内容更丰富
- **A-006**: 系统异常通知仅包括数据源故障、推送通道故障，不包括常规日志
- **A-007**: 飞书卡片按钮的交互（暂停预警/调整阈值）跳转 Dashboard，具体实现由 F5 Dashboard 负责
- **A-008**: MVP 阶段 Telegram 为可选通道，不配置则仅用飞书+本地日志兜底
- **A-009**: 推送日志保留 30 天，超期每日凌晨自动清理
- **A-010**: MVP 飞书卡片仅包含"查看详情"按钮（跳转 Dashboard URL），暂停/调整阈值按钮留到 v1.1
- **A-011**: 飞书 API 限流/错误检测依赖 API 返回的错误码（而非本地计数），返回限流或认证错误时立即标记主通道不可用并降级到 Telegram

## Clarifications

### Session 2026-05-26

- **Q1**: MVP 阶段 Telegram 代理是否为必选项？
  - **A**: Telegram 为可选通道。MVP 阶段不强制要求代理配置，未配置则仅用飞书+本地日志兜底。理由：降低接入门槛，飞书是国内主通道。
- **Q2**: 推送日志保留时长？
  - **A**: 保留 30 天，超期每日凌晨自动清理。理由：足够用户排查近期问题，存储可控。Dashboard 仅展示最近 7 天。
- **Q3**: 飞书卡片按钮交互是否 MVP 实现？
  - **A**: MVP 仅实现"查看详情"按钮（纯 URL 跳转 Dashboard），暂停/调整阈值按钮留到 v1.1。理由：查看详情是最有价值的按钮，实现简单；其他按钮需回调处理逻辑，复杂度较高。
- **Q4**: MVP 阶段简报推送触发源？
  - **A**: 由 quote_scheduler 在交易日 9:00 触发固定模板简报（大盘指数 + 自选股概览）。理由：简报有明确价值，固定模板即可满足，无需等 AI。
- **Q5**: 飞书 API 限流/错误检测策略？
  - **A**: 依赖飞书 API 返回的错误码，返回限流或认证错误时立即标记主通道不可用并降级。理由：无需本地计数器，实现简单，首次触发后即时切换。
