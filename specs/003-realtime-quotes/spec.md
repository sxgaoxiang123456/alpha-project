## User Scenarios & Testing *(mandatory)*

### User Story 1 - 打开 Dashboard 看到最新行情 (Priority: P1)

作为盯盘用户，当我打开 Dashboard 时，我希望立即看到我的自选股列表中每只股票的价格、涨跌幅、成交量等关键信息，且数据是最近几分钟内的。

**Why this priority**: 这是用户打开系统的首要目的——"看行情"。如果 Dashboard 打开后行情数据加载缓慢或过期，用户会立即认为系统不可用。首屏体验直接决定留存率。

**Independent Test**: 单独测试时，系统能在不启动定时任务、不触发预警的情况下，响应 Dashboard 的行情查询请求并返回最新数据。

**Acceptance Scenarios**:

1. **Given** 用户有 20 只自选股，**When** 打开 Dashboard，**Then** 30 秒内展示所有股票最新价格、涨跌幅、成交量，数据时间戳为最近 5 分钟内
2. **Given** 用户无任何自选股，**When** 打开 Dashboard，**Then** 展示"添加自选股"引导，行情区隐藏
3. **Given** 主数据源不可用但备用源可用，**When** 打开 Dashboard，**Then** 展示备用源数据，页面显示"数据来自备用源"提示，不展示错误页
4. **Given** 全部数据源不可用，**When** 打开 Dashboard，**Then** 展示上次缓存数据 + "数据更新延迟"提示，行情区正常展示

### User Story 2 - 定时自动刷新行情 (Priority: P1)

作为盯盘用户，我希望系统每 1-5 分钟自动刷新一次行情，这样即使我不手动刷新页面，也能看到最新的价格变动。

**Why this priority**: 盯盘的核心价值是"自动"。如果用户需要手动刷新才能看到新数据，产品的自动化价值大打折扣。定时刷新也是预警检测的前提（预警需要最新行情来触发）。

**Independent Test**: 单独测试时，启动定时任务后，系统能在配置的时间间隔内自动获取并更新行情数据，无需用户交互或 Dashboard 打开。

**Acceptance Scenarios**:

1. **Given** 系统已配置 3 分钟刷新周期，**When** 交易时段内，**Then** 每 3 分钟自动获取一次全部自选股的最新行情，更新缓存
2. **Given** 定时任务触发时主数据源限流，**When** 执行刷新，**Then** 自动降级到备用源，数据正常更新，标记状态为 "degraded"
3. **Given** 定时任务触发时全部数据源不可用，**When** 执行刷新，**Then** 保留上次缓存数据，推送"数据源异常"通知到用户主通道，标记状态 "unavailable"

---

### User Story 3 - 非交易时段展示收盘数据 (Priority: P2)

作为收盘后查看系统的用户，我希望看到最近收盘时的行情数据，而不是空白或错误页面。

**Why this priority**: 用户会在午休、收盘后、周末查看系统复盘。非交易时段展示收盘数据是基本的可用性要求，避免用户在非交易时段看到空白页面而产生"系统坏了"的误解。

**Independent Test**: 单独测试时，在非交易时段（如 20:00）打开 Dashboard，系统能返回最近收盘数据并明确标注"已收盘"。

**Acceptance Scenarios**:

1. **Given** 当前时间为周六 10:00（非交易日），**When** 用户打开 Dashboard，**Then** 返回最近交易日收盘数据，状态标记为 "market_closed"，页面显示"已收盘"提示
2. **Given** 当前时间为工作日 12:00（午休时段），**When** 用户打开 Dashboard，**Then** 返回上午收盘数据，状态标记为 "market_closed"

---

### User Story 4 - 停牌股票特殊标记 (Priority: P2)

作为持有停牌股票的用户，我希望停牌股票在行情列表中被明确标记，而不是显示为 0 或空白，避免我误以为数据缺失。

**Why this priority**: 停牌是 A 股常见状态（如重组、重大事项）。如果停牌股票显示为价格 0 或空白，用户会误以为是系统 bug 或数据缺失。明确的停牌标记能消除这种困惑。

**Independent Test**: 单独测试时，停牌股票在行情列表中显示停牌标记，其他股票正常展示。

**Acceptance Scenarios**:

1. **Given** 用户自选股中包含一只停牌股票（如 "600519" 停牌），**When** 查看行情列表，**Then** 该股票显示"停牌"标记，不显示涨跌幅和成交量，其余股票正常
2. **Given** 某只股票从停牌恢复交易，**When** 下次定时刷新时，**Then** 该股票正常展示行情数据，停牌标记移除

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供 API 供 Dashboard 主动查询全部自选股的最新行情快照
- **FR-002**: 系统 MUST 通过 APScheduler 定时任务，按配置的周期（1-5 分钟，默认 3 分钟）自动刷新行情数据
- **FR-003**: 系统 MUST 支持查询大盘指数（上证指数、深证成指、创业板指）的最新行情
- **FR-004**: 系统 MUST 每次刷新时获取自选股列表（从 F1 自选股管理模块读取），动态调整监控目标
- **FR-005**: 系统 MUST 对获取的原始数据进行清洗：格式标准化、异常值检测（如价格为负或 0）、缺失值标记
- **FR-006**: 系统 MUST 将清洗后的行情数据写入实时缓存，供 Dashboard 和预警模块读取
- **FR-007**: 系统 MUST 将行情数据异步落盘到历史数据表，保留最近 90 天，超期自动清理
- **FR-008**: 系统 MUST 识别停牌股票并标记状态，停牌股票展示最后交易价格（停牌前收盘价），涨跌幅和成交量展示为"-"
- **FR-009**: 系统 MUST 识别非交易时段（9:30-11:30, 13:00-15:00 之外），返回最近收盘数据，标记 "market_closed"
- **FR-010**: 系统 MUST 向调用方返回数据来源状态（"primary" / "fallback" / "cached" / "unavailable"），以及数据实际时间戳
- **FR-011**: 系统 MUST 支持批量查询 50-100 只股票的行情，p95 响应时间 < 10 秒
- **FR-012**: 系统 MUST 在交易时段内保持 >= 99% 的可用性

### Key Entities

- **Quote（行情快照）**: 代表某只股票在某一时刻的行情数据。核心属性包括：股票代码、股票名称、当前价格、涨跌幅（%）、涨跌额、成交量、成交额、更新时间戳、数据状态（正常/停牌/缺失）。Quote 是调用方（Dashboard、预警）看到的数据形态。
- **MarketIndex（大盘指数）**: 代表大盘指数的行情数据。核心属性包括：指数代码（如 "sh000001"）、指数名称（如 "上证指数"）、当前点位、涨跌幅、涨跌额、成交额、更新时间。大盘指数与自选股分开管理，系统内置监控列表。
- **HistoricalQuote（历史行情）**: 代表一只股票的历史行情记录，用于简报生成和回测。核心属性包括：股票代码、日期、开盘价、收盘价、最高价、最低价、成交量、成交额。HistoricalQuote 异步落盘，不影响实时查询性能。
- **QuoteCache（行情缓存）**: 代表最新行情的缓存条目。核心属性包括：股票代码、缓存内容（JSON 序列化的 Quote）、缓存时间、过期时间。复用 F2 数据多源容灾的 CacheEntry 机制，但 QuoteCache 的过期时间更短（5 分钟）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Dashboard 首屏加载（含行情数据渲染完成）p95 < 3 秒
- **SC-002**: 20 只自选股批量查询的 p95 响应时间 < 10 秒
- **SC-003**: 单只股票查询的 p95 响应时间 < 3 秒
- **SC-004**: 定时刷新周期稳定运行，实际间隔与配置值偏差 < 10%
- **SC-005**: 行情数据时间戳与查询时间的差值 < 5 分钟（表示数据新鲜度）
- **SC-006**: 停牌股票识别准确率 = 100%（数据源明确返回停牌状态的必须标记）
- **SC-007**: 交易时段内行情刷新成功率 >= 99%（含降级场景）
- **SC-008**: 历史数据异步落盘延迟 < 1 分钟（从获取到写入数据库）

## Clarifications

### Session 2026-05-26

- **Q1**: MVP 阶段历史数据落盘的范围？
  - **A**: 落盘但滚动保留 90 天（Option B，recommended）。MVP 实现异步落盘验证技术可行性，超期数据自动清理。保留 90 天数据支撑 v1.1 简报测试。FR-007 更新为"保留最近 90 天"，SC-008 保留。
- **Q2**: MVP 阶段交易日历判断策略？
  - **A**: 运行时查询 AkShare 交易日历（Option A，recommended）。每次定时任务触发前查询 AkShare 判断今天是否交易日。数据源不可用时默认按工作日判断（简化降级）。A-008 已更新。
- **Q3**: 停牌股票的价格是否展示？
  - **A**: 展示最后交易价格，涨跌幅和成交量留空（Option A，recommended）。价格字段显示停牌前收盘价，涨跌幅和成交量显示为"-"或留空。FR-008 已更新。

## Assumptions

- **A-001**: 大盘指数固定监控 3 个：上证指数（sh000001）、深证成指（sz399001）、创业板指（sz399006），不支持用户自定义增减
- **A-002**: 行情数据通过 F2 数据多源容灾的 facade 层获取，本模块不直接调用 AkShare/BaoStock
- **A-003**: 自选股列表从 F1 的 WatchlistItem 表读取，本模块不维护独立的监控列表
- **A-004**: 定时刷新周期通过配置文件或环境变量设置，默认 3 分钟，交易时段内生效，非交易时段暂停定时刷新
- **A-005**: 历史数据落盘使用异步任务（如 asyncio 或后台线程），不阻塞实时查询响应
- **A-006**: 行情缓存复用 F2 的 CacheEntry 机制，但过期时间单独配置为 5 分钟（与 F2 的 1 小时不同）
- **A-007**: 数据清洗规则：价格为负或 0 标记异常；涨跌幅绝对值 > 20%（科创板/创业板 > 30%）标记异常；成交量为 0 且非停牌标记异常
- **A-008**: A 股交易时段为工作日 9:30-11:30、13:00-15:00。系统通过 AkShare 交易日历接口实时查询当日是否为交易日，数据源不可用时降级为"周一到周五为交易日"的简化假设
